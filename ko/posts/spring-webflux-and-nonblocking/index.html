Spring WebFlux와 논블로킹에 대한 생각

Dec 20, 2020
10 minute read

그동안은 Spring MVC만 계속 써 왔기 때문에, 요즘은 Kotlin과 Spring WebFlux를 함께 써 보면서 간단한 앱을 만들어 보고 있습니다. Spring WebFlux 자체는 이전 글에서 이미 소개했으니 여기서는 자세한 설명은 생략하겠습니다. 나온 지 꽤 시간이 지났지만, 아직 현업에서 널리 퍼졌다고 보기는 어렵고 관련 자료도 생각보다 많지 않습니다.

아마도 도입 장벽이 아직 높기 때문일 겁니다. 프레임워크 자체가 완전히 안정된 인상도 아니고, 핵심이 되는 Reactor 스타일에 익숙하지 않은 사람도 많습니다. Reactive라고 해도 RxJava와는 결이 조금 달라서, 배우는 데 드는 비용이 낮다고 말하기도 어렵습니다. 회사 입장에서도 새 기술을 바로 들이기에는 리스크가 있고, 엔지니어의 학습 비용도 무시하기 어려워서 선뜻 장점만 보이지는 않습니다.

성능도 문제입니다. 기존 Spring MVC 프로젝트를 WebFlux로 바꾸는 것만으로는 기대만큼 큰 이득이 나오지 않는 경우가 많습니다. 신규 프로젝트라면 어떨까 싶지만, 스타트업이나 벤처에서는 애초에 JVM 언어 자체를 잘 쓰지 않는 경우도 적지 않습니다. 제 편견일 수도 있지만, 그런 환경에서는 Python, Ruby, JavaScript가 더 흔하게 보입니다. JVM에 익숙한 엔지니어가 있더라도, 기술이 아직 충분히 검증되지 않았고 학습 비용도 높다는 문제는 그대로 남습니다.

그래서 적어도 엔터프라이즈 환경에서는 여전히 Spring WebFlux를 쉽게 도입하기 어렵다고 봅니다. 다만 Spring의 미래가 WebFlux에 있다는 이야기도 많고, 비동기와 동시 처리 성능이 점점 더 중요해지는 흐름도 있기 때문에, 개념은 미리 익혀 둘 필요가 있습니다. 직접 WebFlux로 앱을 만들어 보면서 느낀 점을 정리해 보겠습니다.

Spring WebFlux 앱을 작성한다는 것

Spring WebFlux는 기존의 Controller + Service 패턴으로도 충분히 코드를 작성할 수 있습니다. 겉으로는 기존 Spring MVC 프로젝트를 그대로 두고 일부만 바꾸거나, 적은 수정으로 쉽게 옮길 수 있을 것처럼 보입니다. 실제로는 생각보다 그렇지 않았습니다.

먼저 간단한 CRUD 샘플(GitHub 저장소)을 만들고, 이를 바탕으로 조정씨 비슷한 서비스를 만들어 보려 했습니다. 샘플은 Spring Data JPA를 쓰고 있었지만, WebFlux에서는 본격적으로 논블로킹 구조를 써 보고 싶어서 R2DBC를 도입했고 DB 접근도 비동기로 바꿨습니다. 비동기 흐름 안에 동기 처리가 하나라도 끼면 전체가 동기로 돌아가기 쉬우니, ORM도 그에 맞춰 R2DBC로 맞추는 편이 낫다고 판단했습니다.

다행히 R2DBC는 Spring Data JPA나 Spring Data JDBC와 비슷한 감각으로 사용할 수 있었습니다. 인터페이스 형태의 Repository를 만들고 DTO 형태의 객체만 두어도 DB 매핑이 가능합니다. Kotlin에서는 data class만으로도 충분했습니다. JPA보다 어노테이션도 적어서 테이블 객체를 정의하는 것도 단순합니다. 메서드 이름만으로 쿼리를 만들어 주는 점도 처음에는 꽤 편했습니다. 다만 처음 쓰는 기술인 만큼, 생각처럼 모든 것이 순조롭지는 않았습니다.

Join이 안 된다

아무리 좋은 기술이라도 안정화되기 전에는 “기존 기술을 바로 대체할 수 있다"는 확신을 주기 어렵습니다. R2DBC도 마찬가지로, 기존 ORM을 당장 밀어낼 만큼 편한 점과 불편한 점이 함께 있습니다. 그중 가장 큰 차이는 자동 Join을 다루기 어렵다는 점입니다. 즉, 테이블 간 관계를 미리 객체로 정의하는 방식이 잘 맞지 않습니다.

Spring Data JPA나 JDBC에서는 어노테이션만으로 @OneToOne 같은 관계를 쉽게 정의할 수 있고, 코드에서도 그 관계가 자연스럽게 드러납니다. 하지만 R2DBC는 아직 이런 부분이 약합니다. 그래서 Repository에 @Query를 붙여 직접 Join SQL을 쓰거나, 애플리케이션 안에서 두 객체를 합쳐야 합니다.

저는 전자는 코드 커버리지와 유지보수 측면에서 아쉬워서, 후자를 선택했습니다. 테이블과 객체를 1:1로 맞춰 두면 구조가 바뀌었을 때 수정 범위도 단순해집니다. 그런데 여기서 또 다른 문제가 있었습니다. R2DBC의 결과가 일반 객체가 아니라 Mono나 Flux였기 때문입니다.

block()의 딜레마

Repository가 돌려주는 값이 Mono나 Flux라면, 두 객체를 합치는 방식부터 다시 생각해야 합니다.

가장 쉬운 방법은 block()으로 값을 꺼내서 동기적으로 쓰는 것입니다. Mono와 Flux에는 이미 block() 메서드가 있으니 코드만 보면 어렵지 않습니다. Spring MVC에서도 RestTemplate보다 WebClient를 쓰는 편이 낫다고 자주 말하듯, 동기와 비동기가 완전히 배타적인 것은 아닙니다.

문제는 그렇게 하면 비동기의 장점이 크게 줄어든다는 점입니다. 중간에 블로킹이 끼는 순간 동시성 이점이 약해지기 때문입니다. 결국 그렇게 쓸 거라면 굳이 WebFlux나 R2DBC를 선택할 이유가 약해집니다. 그래서 다른 방식을 찾아야 했습니다.

Mono + Mono 또는 Flux + Flux

결국 비동기 방식에 맞춰 처리하는 수밖에 없었습니다. 찾아보니 두 개의 Mono나 Flux를 묶어 다루는 방법이 있었고, 핵심은 zipWith()였습니다. 합쳐진 값은 Kotlin에서 튜플처럼 다룰 수 있으니 map()에서 t1, t2를 꺼내 쓰면 됩니다. 예시는 다음과 같습니다.

fun getParticipant(participantId: Long): Mono<ParticipantDto> =
    repository.findById(participantId)
        .zipWith(candidateParticipantHandler.getCandidateParticipantsByParticipantsId(participantId).collectList())
        .map { mapDto(it.t1, it.t2) }

다만 이런 방식으로 Join하려면, 애초에 두 개의 Mono나 Flux를 따로 가져올 수 있어야 합니다. 단순 조회 API라면 큰 문제가 없지만, 데이터 구조가 복잡해질수록 이 방식은 점점 불편해집니다.

테이블을 Join할 때의 한계

특히 List를 결과로 돌려줘야 하는 경우가 불편했습니다. 예를 들어 참가자 목록을 뽑고, 그 참가자에 대응하는 후보자 정보를 다시 Join해야 한다면 다음 순서가 필요합니다.

  1. 키를 기준으로 원본 테이블을 조회해 Mono 또는 Flux로 가져온다.
  2. 얻은 Mono 또는 Flux에서 map()으로 추가 키를 꺼내고, Join 대상 테이블을 다시 조회한다.
  3. 원본 데이터와 대상 데이터를 합친다.

겉보기에는 그럴듯하지만, 두 번째 단계에서 다시 Mono나 Flux를 얻는 순간 원래 객체를 꺼내는 과정이 애매해집니다. 이때 또 block()을 쓰면 지금까지 비동기로 이어 온 흐름이 무너집니다. 그래서 이런 방식의 Join은 현재로서는 꽤 불편하고, 사실상 마땅한 대안도 잘 보이지 않았습니다.

이게 최선인가

이렇게 WebFlux를 만지다 보면 문제를 하나씩 풀 수는 있습니다. 다만 저는 이런 방식 자체가 꽤 어색하게 느껴졌습니다. 그 이유는 Project Loom 리더 Ron Pressler의 설명에서 어느 정도 찾을 수 있었습니다. 그의 말에 따르면 현재의 비동기 프로그래밍에는 세 가지 문제가 있습니다.

제어 흐름을 놓치기 쉽다

비동기 방식으로 코드를 쓰다 보면, 정작 어떤 논리와 목적을 위해 작성한 코드인지 흐려질 때가 있습니다. 비동기 코드를 쓰다 보면 비즈니스 로직보다 비블로킹 문법 자체에 더 신경을 쓰게 되기 때문입니다. 조건 분기나 반복 같은 단순한 구조도 비동기로 바꾸면 코드가 복잡해지고, 전체 흐름을 놓치기 쉬워집니다. Java만 써 온 제게는 덜 와닿았지만, JavaScript로 비동기 코드를 써 본 분이라면 쉽게 이해할 수 있을 겁니다. 콜백 헬 같은 문제가 대표적이죠.

예를 들어 이 글에서 다룬 것처럼 두 테이블을 Join한다고 해 봅시다. 동기 코드라면 처음부터 Join된 데이터를 가져오거나, 객체 두 개를 순서대로 선언해 처리하면 됩니다. 그런데 비동기로 바꾸는 순간, “무엇을 하려는가"보다 “비동기 방식에 맞게 어떻게 쓸 것인가"가 먼저가 되어 버립니다. 결국 원래 목적을 흐리기 쉬워집니다.

컨텍스트를 잃어버린다

비동기에서는 예외가 났을 때 스택 추적이 매우 어렵습니다. 동기 처리에서는 보통 하나의 스레드가 하나의 요청을 처리하므로, 어떤 코드가 실행됐는지 그 스레드의 히스토리만 봐도 충분합니다. 하지만 비동기에서는 하나의 요청이 여러 스레드를 오가며 처리되기 때문에, 한 스레드의 이력만으로는 전체 상황을 파악하기 어렵습니다.

코드가 전염된다

비동기 코드로 작성하기 시작하면, 결국 애플리케이션 전체에서 동기라는 개념을 걷어내게 될 수 있습니다. 앞서 말했듯이 비동기 흐름 안에 동기 처리가 하나만 섞여도 전체가 블로킹으로 돌아가기 때문입니다. 그래서 하나의 앱 안에서 동기와 비동기를 함께 쓰는 일이 점점 어려워지고, 비동기 코드가 다른 코드까지 “전염"되는 상황이 생깁니다.

WebFlux에서는 동기 코드와 섞어 쓰는 것이 거의 불가능에 가깝기 때문에, 결국 객체를 Mono나 Flux로 감싸고, zipWith()로 연결하고, map()이나 flatMap()으로 처리하는 방식으로 갈 수밖에 없습니다. 반대로 block()으로 값을 꺼내기 시작하면, 처음부터 동기로 쓰는 편이 낫다는 결론에 가까워집니다.

그래도 비동기는 필요하다

이런 문제는 비동기를 써 본 사람이라면 한 번쯤 겪어 봤을 이야기입니다. 그렇다고 비동기 프로그래밍의 필요성까지 사라지는 것은 아닙니다. 특히 지금처럼 많은 Web 애플리케이션에서 동시 처리 성능이 중요한 상황이라면 더 그렇습니다. 실제로 Google Analytics는 KissMetrics를 인용해 페이지 응답이 1초 느려지면 전환율이 7% 감소하고, 47%의 사용자는 웹페이지가 2초 이내에 로드되기를 기대한다고 설명합니다. 이런 요구에 대응하려고 JavaScript와 C#은 async/await/promise를 발전시켰고, Kotlin은 Coroutine을 도입했습니다.

특히 Java는 OS 스레드를 직접 사용하기 때문에 동시에 처리할 수 있는 요청 수에 한계가 있습니다. 이런 스레드 기반 언어의 한계를 넘기 위해 여러 비동기 라이브러리가 나왔지만, 라이브러리만으로는 한계가 있어서 JVM 레벨의 대응이 논의되고 있습니다. 그게 바로 Project Loom입니다.

Project Loom이라는 답

Project Loom은 기존 스레드를 Fiber라는 가벼운 가상 스레드로 나눠 동시 처리 성능을 높이고, 비동기 프로그래밍을 동기 프로그래밍처럼 다룰 수 있게 하는 것을 목표로 합니다. 그렇게 되면 지금처럼 비동기와 동기에 따라 코드가 완전히 달라지는 문제를 줄일 수 있습니다. Fiber는 매우 많이 생성할 수 있고, 기존 스레드와 사용 방식도 크게 다르지 않아서 코드 수정도 적습니다. 또 내부적으로는 async/await를 쓰는 것처럼 처리되므로 콜백 헬 같은 문제도 완화됩니다.

Kotlin의 Coroutine은 컴파일러 레벨의 대응이기 때문에 앞서 말한 문제 중 컨텍스트를 잃는다와 코드가 전염된다는 부분을 근본적으로 해결하긴 어렵습니다. 반면 Project Loom은 JVM 레벨의 변화라서 이런 문제를 전반적으로 줄여 줄 수 있다는 장점이 있습니다. 기존 동기 API를 비동기로 바꾸더라도 클라이언트 코드 변화가 크지 않다는 점도 매력적입니다.

많은 프로그래머가 동기적인 작성 방식에 익숙한 현실을 생각하면, Project Loom이 정식 릴리스될 경우 기존의 Coroutine이나 async/await/promise, Reactive Stream 기반 프로젝트의 상당수가 Fiber 기반으로 옮겨갈 가능성도 있습니다.

마지막으로

어떤 분야든 과도기는 가장 혼란스럽고 힘든 시기입니다. 프로그래밍에서도 지금의 비동기 프로그래밍이 딱 그런 상태에 가깝다고 생각합니다. 많은 사람들이 언어, 라이브러리, 프레임워크 여러 방향에서 비동기의 단점을 줄이려고 노력해 왔고, 이제는 그 결과가 조금씩 보이는 것 같습니다. Reactive Stream이나 반응형 프로그래밍 자체는 분명 의미가 있지만, 기존 방식과 너무 달라 학습 비용이 크다는 점도 무시하기 어렵습니다.

다만 Project Loom은 아직 개발 중인 기술이기 때문에, 나중에 한계나 다른 문제가 드러날 가능성도 있습니다. WebFlux 같은 곳에 채택되더라도 실제 릴리스 시점을 예측하기 어렵고, 그전까지는 Reactive Stream이나 Coroutine을 계속 써야 합니다. Loom이 나온다고 해서 기존 코드를 한꺼번에 다시 쓰는 것도 현실적이지 않습니다.

그럼에도 JVM 자체가 바뀌고 비동기 프로그래밍을 지금보다 더 자연스럽게 다룰 수 있게 된다면, 그것만으로도 꽤 큰 변화일 것입니다. 요즘은 Kotlin 쪽에 더 관심이 가지만, Java와 JVM이 이 문제를 어떻게 풀어 가는지 지켜보는 일도 여전히 흥미롭습니다.
