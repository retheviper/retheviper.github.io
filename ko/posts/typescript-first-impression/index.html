TypeScript를 써 본 첫인상

May 16, 2020
9 minute read
日本語

이번에 새 프로젝트에서 Angular와 Spring Boot로 웹 애플리케이션을 개발하게 됐습니다. Spring Boot는 계속 써 왔기 때문에 크게 걱정되지 않았지만, 문제는 Angular였습니다. Angular는 버전 2부터 TypeScript를 본격적으로 사용한다고 들었고, 저는 JavaScript와 jQuery를 조금 만져 본 정도라 TypeScript가 정확히 어떤 언어인지도 잘 몰랐습니다. 그때 제가 알고 있던 것은 “Java나 C#처럼 정적인 느낌으로 코드를 쓰면, 그것을 적당히 컴파일해서 JavaScript로 바꿔 주는 언어"라는 정도였습니다.

그래서 우선 TypeScript부터 공부해 보기 시작했는데, 언어 자체의 특징도 그렇고 실제로 TypeScript로 무언가를 만들 때 필요한 지식의 범위도 생각보다 넓다는 걸 알게 됐습니다. 이번 글에서는 Java밖에 잘 모르던 초보자의 입장에서, TypeScript를 처음 접하며 느낀 점과 무엇을 알아야 하는지를 솔직하게 정리해 보려고 합니다.

놀랄 만큼 Java와 비슷하다

원래 C/C++, C#, Java 같은 이른바 C 계열 언어와 비슷하다는 이야기는 들었습니다. 그런데 실제로 TypeScript를 조금 만져 보니 정말 그랬습니다. 처음에는 단순히 “변수 선언에 타입을 붙일 수 있는 JavaScript” 정도로 생각했는데, 막상 써 보니 그 이상으로 객체 지향 스타일에 맞춰 발전했거나, 어떤 면에서는 다시 돌아온 느낌도 있었습니다.

타입 표기가 꽤 괜찮다

타입 시스템은 원래 동적 언어였던 JavaScript를 정적인 방식으로 다루게 만들어 줍니다. 그렇다고 문법이 낡은 느낌은 아니고, 오히려 Kotlin이나 Swift처럼 비교적 현대적인 스타일에 가깝습니다. 타입을 먼저 적는 것이 아니라, 변수를 선언한 뒤 타입을 붙이는 방식1이기 때문입니다.

Java에서는 이렇게 씁니다.

int a = 1000;

TypeScript는 이렇게 씁니다.

var a: number = 1000;

TypeScript에서는 이런 타입 표기를 “type annotation"이라고도 부릅니다. 어차피 최종적으로는 런타임 타입이 사라진 JavaScript로 컴파일되기 때문에, 일종의 컴파일러를 위한 힌트나 주석처럼 보는 느낌에 가깝다고 할 수 있겠습니다.

타입 외에도 닮은 점이 많다

TypeScript에는 타입 표기만 있는 것이 아닙니다. Java나 C# 개발자에게 익숙한 access modifier, interface, class, generics, decorator(어노테이션 비슷한 개념) 같은 요소도 들어 있습니다. 이 중 일부는 최근 JavaScript에서도 지원되긴 하지만, 실제 서비스 개발에서는 브라우저 호환성 때문에 최신 JavaScript 기능을 마음껏 쓰지 못하는 경우도 많습니다.

그런 점에서 TypeScript는 컴파일 옵션을 통해 어떤 버전의 JavaScript로 출력할지를 정할 수 있기 때문에, 브라우저를 과하게 신경 쓰지 않고도 비교적 익숙한 형태로 코드를 작성할 수 있다는 장점이 있습니다. 이건 실제로 꽤 반가웠습니다.

또 의존성을 npm 같은 패키지 매니저로 관리할 수 있다는 점도 좋았습니다. 물론 이건 JavaScript에서도 원래 가능한 일이지만, import와 함께 쓰면 거의 C 계열 언어처럼 운영할 수 있다는 감각이 있습니다. JavaScript의 모듈 시스템도 ES6 이후에는 많이 정리됐지만, 그 이전 환경까지 고려해야 한다면 여전히 꽤 번거로울 수 있겠다는 생각도 들었습니다. 물론 제가 아직 익숙하지 않아서 그렇게 느꼈을 가능성도 있습니다.

조금 걸렸던 부분

TypeScript의 가장 큰 장점은 역시, 전통적인 정적 언어에 익숙한 사람에게 꽤 쾌적한 개발 환경을 제공하면서 컴파일 타임에 오류를 많이 잡아 준다는 점이라고 생각합니다. 다만 공부하면서 몇 가지 걸리는 부분도 있었기 때문에 그것도 같이 적어 보겠습니다.

결국은 JavaScript다

TypeScript는 JavaScript의 superset이기 때문에 JavaScript 문법을 그대로 사용할 수 있다는 점이 큰 특징으로 자주 언급됩니다. 처음에는 저도 이걸 장점으로 봤습니다. 그런데 공부를 하다 보니, 이것이 단점이 되기도 한다는 생각이 들었습니다.

JavaScript를 그대로 쓸 수 있다는 말은, 바꿔 말하면 하나의 코드베이스 안에 순수 JavaScript와 TypeScript가 섞여 있어도 기술적으로는 문제가 없다는 뜻이기도 합니다. TypeScript가 superset이라는 특성을 유지하는 한, 이 자체로 코드는 동작할 수 있습니다. 하지만 사람 입장에서는 오히려 굉장히 혼란스러운 코드가 생길 여지가 있습니다.

JavaScript만으로도 쓸 수 있다는 것의 의미

애초에 JavaScript는 TypeScript보다 역사가 훨씬 길고, 사용자 수도 압도적으로 많습니다. 그리고 저처럼 C 계열 언어에서 출발한 사람이 아니라, JavaScript로 프로그래밍을 시작한 사람도 많습니다. 그런 사람들에게 TypeScript는 “굳이 써야 할 이유는 잘 모르겠고, 약간 불편한 것"으로 느껴질 수도 있습니다.

또 JavaScript에 익숙한 사람이 TypeScript로 넘어와도, 결국 JavaScript 스타일에서 크게 벗어나지 않는 코드를 계속 쓸 가능성도 충분히 있습니다.

TypeScript가 superset으로 설계된 것은 기존 JavaScript 개발자를 흡수하기 위한 전략이었을 가능성이 크다고 봅니다. 하지만 저는 프로그래밍 언어가 자유도가 높을수록 혼란도 커질 수 있다고 생각하기 때문에, 이 특성은 분명 장점이면서 동시에 단점이기도 하다고 느꼈습니다.

prototype

제가 클래스 기반 객체 지향 언어에서 출발해서 그런지, JavaScript의 prototype 개념은 아직도 완전히 익숙하지 않습니다. JavaScript에는 prototype 기반이라는 특성이 있고, 객체를 훨씬 자유롭게 다룰 수 있습니다2. 그런데 Java 프로그래머 입장에서는 “객체는 클래스에서 만들고, 클래스는 보통 파일 단위로 존재한다"는 감각이 강하기 때문에, 이런 자유도가 오히려 혼란스럽게 느껴질 수 있습니다.

물론 이건 TypeScript보다는 JavaScript 쪽의 특성입니다. 하지만 TypeScript가 JavaScript의 superset인 이상, 결국 완전히 피할 수는 없는 부분이기도 합니다.

모듈

모듈 시스템도 흥미롭지만 동시에 까다로운 부분이었습니다. 파일 하나에 클래스 하나씩 두고 import해서 쓰다 보면 얼핏 Java와 크게 다르지 않아 보이는데, 실제로는 세부 동작이 꽤 복잡한 느낌이 있었습니다.

예를 들어 Python의 as처럼 alias를 붙이는 방식도 있고, import 스타일에 따라 동작이 달라지는 경우도 있으며, 환경에 따라 CORS 설정이 걸리는 경우도 있다고 들었습니다. 아직 깊이 써 보진 않았지만, “실제로 써 보기 전에는 감이 잘 안 온다"는 점이 꽤 신경 쓰였습니다.

// 두 가지 문법이 가능하지만 같은 의미는 아니다
import express from "express";
import * as express from "express";

컴파일

TypeScript를 설치하면 tsc 명령을 사용할 수 있고, .ts 파일을 .js로 컴파일할 수 있습니다. tsconfig.json으로 컴파일 대상, 출력 위치, 타깃 JavaScript 버전 같은 여러 옵션도 지정할 수 있습니다.

다만 컴파일이 생각보다 느리다는 인상이 있었고, 결국 결과물은 어디까지나 JavaScript입니다3. 즉, TypeScript로 타입을 잘 적어 둔다고 해도 런타임에서는 다시 느슨한 JavaScript 환경으로 들어가게 됩니다. any를 쓰지 않고 타입을 꼼꼼히 적는 것만으로는 충분하지 않을 수 있고, 런타임에서도 타입 관련 문제가 생기지 않도록 신경 써야 할 수 있겠다는 느낌을 받았습니다.

요즘은 npm 대체 런타임처럼 보이는 Deno 같은 것도 등장했지만, 내부적으로는 여전히 tsc에 기대는 부분이 있고, 컴파일 속도 문제 역시 자주 언급됩니다. Rust로 새로운 컴파일러를 만들고 있다는 이야기도 있지만, 언제 안정화될지는 알 수 없습니다.

또 tsc만이 아니라 webpack, ts-loader, Babel 같은 도구를 함께 써야 하는 경우도 있어서 초기 설정이 꽤 번거롭게 느껴졌습니다. 언어를 공부하는 것도 모자라, 컴파일과 번들링을 위해 또 다른 도구 체인을 배워야 한다는 점은 조금 부담스러웠습니다. 어쩌면 JavaScript 생태계와 강하게 연결된 언어로서 피할 수 없는 운명일지도 모르겠습니다.

깔끔하다고 느낀 부분

JavaScript 라이브러리를 그대로 쓸 수 있다

“TypeScript에서도 JavaScript 라이브러리를 사용할 수 있다"는 이야기는 들었지만, 처음에는 그게 어떻게 가능한지 잘 감이 오지 않았습니다. TypeScript는 타입 추론이 있긴 해도, 결국 많은 경우에는 타입 정보를 명시적으로 줘야 잘 작동합니다. 그런데 기존 JavaScript 라이브러리들이 모두 TypeScript를 고려해서 만들어졌을 리는 없으니까요.

이 부분의 답은 생각보다 단순했습니다. .d.ts 형태의 “타입 정의 파일"을 제공하면 된다는 것이었습니다. 이미 TypeScript를 지원하는 라이브러리도 이런 타입 정의 파일을 통해 타입 정보를 제공하고 있고, 직접 만들 수도 있습니다. 그리고 유명 라이브러리의 경우에는 node_modules에서 바로 설치 가능한 형태로 타입 정의 파일을 별도로 제공하는 경우도 많았습니다.

예를 들어 Node의 타입 정의가 필요하다면 아래처럼 설치할 수 있습니다.

npm install --save-dev @types/node

또 GitHub의 DefinitelyTyped 리포지토리를 보면 어떤 라이브러리들의 타입 정의가 제공되는지도 확인할 수 있습니다.

이 구조는 개인적으로 꽤 깔끔하다고 느꼈습니다. JavaScript의 방대한 기존 생태계를 버리지 않으면서도, TypeScript 쪽 타입 안정성을 얹을 수 있기 때문입니다.

마지막으로

아직도 신경 쓰이는 지점은 꽤 있습니다. 하지만 그중 상당수는 TypeScript 자체의 문제라기보다 JavaScript 생태계에서 비롯된 문제처럼 느껴졌습니다. 그래서 “프런트엔드 개발자가 지금 어떤 언어를 고르는 게 좋을까?“라는 질문에는, 적어도 현시점에서는 TypeScript 쪽이 더 낫지 않을까 생각합니다.

물론 제 출발점이 Java였기 때문에 더 그렇게 느낄 수도 있습니다. 그래도 컴파일 타임에 많은 문제를 미리 잡아낼 수 있다는 장점은 쉽게 포기하기 어렵습니다.

지금은 JavaScript 호환성 때문에 초기 설정이 번거롭거나, 컴파일이 느리거나, 도구 체인이 복잡하다는 단점이 분명 있습니다. 하지만 Angular처럼 처음부터 TypeScript를 전제로 설계된 프레임워크와 라이브러리가 더 많아지고, 브라우저나 런타임 환경도 조금씩 정리되면 이런 문제는 점점 줄어들지 않을까 기대하게 됩니다.

처음에는 단순히 Angular를 쓰기 위해 공부를 시작했지만, 생각보다 언어 자체가 잘 정리돼 있고 앞으로도 꽤 유망하다는 인상을 받았습니다. 그래서 엔지니어들에게 한 번쯤은 꼭 추천해 보고 싶은 언어라는 생각이 듭니다.

아, Python도 타입 표기를 조금 더 본격적으로 쓸 수 있으면 좋겠다는 생각을 했는데, 나중에 찾아보니 Python도 3.6부터 타입 선언을 지원하더군요.4
