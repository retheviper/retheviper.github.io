Oracle Java SE 8 Silver 후기

Oct 06, 2019
6 minute read
日本語

이번에 Oracle 인증 Java SE 8 Silver를 봤습니다. 일 때문에 잠시 Java를 쓰게 되면서 제 실력이 어느 정도인지 확인해 보고 싶었고, 운 좋게 합격했습니다. 이미 합격 후기나 공부법 글은 많아서 그런 내용은 여기서 길게 다루지 않겠습니다.

그래도 자격시험 이야기를 굳이 적는 이유는, 시험에서 묻는 내용과 실제 업무에서 필요한 지식 사이에 생각보다 간격이 있기 때문입니다. 시험을 먼저 따고 업무에 들어가는 사람도 있고, 저처럼 일을 하면서 익힌 뒤 시험을 보는 사람도 있습니다. 상황은 조금씩 다르더라도, 시험에서 헷갈리는 지점은 꽤 비슷하다고 생각합니다.

이 글은 JavaSE Silver를 준비하는 분들께, 제가 특히 자주 헷갈렸던 부분을 정리한 것입니다.

전체 인상

업무에서는 잘 쓰지 않는 지식도 적지 않았지만, 그렇다고 전부 쓸모없는 내용은 아니었습니다. 한 번 익혀 두면 실무에서도 쓸 수 있는 내용이 꽤 있었습니다. 문제 유형은 크게 API 자체를 묻는 문제와, 코드를 읽고 결과를 이해하는 문제로 나뉘었습니다.

의외로 까다로웠던 건 IDE에 의존할 수 없다는 점이었습니다. import를 직접 써야 하고, 컴파일 에러를 스스로 찾아야 합니다. 실무에서는 IDE 없이 Java를 작성하는 경우가 거의 없겠지만, 시험 문제에서는 이런 코드가 꽤 자주 나오므로 꼼꼼히 읽어야 합니다.

아래에서는 세부 항목별로 느낀 점을 정리해 보겠습니다.

데이터 타입

가장 많이 쓰는 데이터 타입은 int, double, String 정도입니다. 시험에서 byte나 float의 범위를 묻는 문제는 많지 않았지만, switch에서 쓸 수 없는 타입을 고르는 식의 문제는 나올 수 있습니다. Bronze보다 Silver가 더 쉬운 건 아니므로, Silver부터 시작하는 사람은 조금 헷갈릴 수 있습니다.

또 하나는 타입 변환입니다. 암시적 변환과 명시적 변환을 구분해야 하는데, byte, short, long, float의 특성을 정확히 알아 두지 않으면 의외로 답을 고르기 어렵습니다.

String / StringBuilder

String과 StringBuilder 관련 문제가 꽤 많았습니다. 평소에는 잘 의식하지 않지만, String은 immutable이라 replace나 substring을 써도 새 인스턴스가 반환됩니다. 예를 들어 아래 문제는 출력값을 묻습니다.

// 출력되는 문자열을 맞히는 문제

String s = "this is string";
s.substring(0, 1);
System.out.println(s);

원래 String은 바뀌지 않으므로 "this is string"이 출력됩니다. 하지만 코드를 대충 읽으면 substring 결과가 출력될 것처럼 착각하기 쉽습니다.

배열

실무에서는 배열보다 List를 더 자주 쓰다 보니 이 부분도 꽤 헷갈렸습니다. 배열 선언 방식이나 요소를 다루는 방법을 묻는 문제가 자주 나왔고, 선언 방법도 다양해서 선택지가 은근히 까다로웠습니다.

물론 List는 기본적으로 가변이고 스레드 세이프도 아니어서, 경우에 따라 배열을 쓰는 일이 아예 없는 것은 아닙니다. 어쨌든 알고 있으면 손해는 아니니 확실히 정리해 두는 편이 좋습니다.

접근 제한자 / 상속

default와 protected의 차이를 묻는 문제도 있습니다. 평소 패키지 구조나 상속을 고려하며 코드를 짜지 않으면 public과 private만 쓰다가 다른 제한자를 잊기 쉽습니다.

상속 문제에서는 접근 제한자가 핵심이 되는 경우가 많습니다. 예를 들어 슈퍼클래스 메서드를 더 좁은 범위로 오버라이드한 코드가 주어지고, 실행 결과를 묻는 식입니다. 실행될 것 같아도 정답은 컴파일 에러일 수 있습니다.

상속 관련 문제는 양도 많고 종류도 다양했습니다. 인터페이스와 추상 클래스의 차이, 상속 방식, 인스턴스와 참조의 차이, 캐스트 같은 주제가 자주 나왔습니다. 실무에서 자주 보던 내용이라도 시험에서는 꼼꼼하게 읽어야 했습니다.

라벨

라벨은 이번에 처음 제대로 접했습니다. 이중 루프에서 쓰면 유용할 수 있어서, 시험용이 아니더라도 알아 두면 좋습니다. 다만 루프 밖에서 쓸 일은 많지 않을 것 같습니다.

문제는 라벨이 붙은 루프와 continue/break의 동작을 함께 묻는 형태가 있었습니다. 예를 들면 아래와 같습니다.

// 출력 결과를 묻는 문제

int num = 0;

x:
for (int i = 0; i < 10; i++) {
    if (i % 5 == 0) {
        continue x;
    }

    System.out.println(num);
    num++;
}

라벨 자체보다 continue와 break의 차이를 이해하고 있는지를 묻는 문제라고 보는 편이 맞습니다. 그래도 라벨 개념을 모르면 바로 막힐 수 있습니다.

예외

Java 예외의 특징과 try-catch 동작을 묻는 문제도 나옵니다. 실무에서 커스텀 예외를 쓰는 경우가 있다면 Exception과 RuntimeException의 차이는 기본적으로 알아 둬야 합니다. 또 throws 선언이 있는지, catch 순서가 맞는지 같은 부분도 자주 헷갈립니다. 상위 예외를 먼저 잡으면 하위 예외는 뒤에서 다시 쓸 수 없다는 점도 주의해야 합니다.

IDE 없이 코드를 보면 놓치기 쉬운 부분이라, 시험에서는 더 조심해야 합니다.

Lambda

Java SE Silver에서는 Java 8 전용 API가 꽤 나옵니다. Lambda도 그중 하나입니다. 대부분은 올바른 문법이나 Function 계열 타입을 묻는 문제라서, 익숙하지 않아도 규칙만 외워 두면 충분합니다.

특히 괄호나 return 생략이 맞는지 확인하는 문제는 IDE라면 바로 잡아 주지만, 시험에서는 직접 판단해야 해서 실수하기 쉽습니다.

LocalDate / LocalTime / LocalDateTime

제가 기존에 주로 쓰던 날짜 API는 java.util.Date와 java.sql.Date였기 때문에 이 부분도 꽤 어려웠습니다. 이들도 String처럼 immutable이라, 메서드가 새 인스턴스를 반환한다는 점을 알아야 합니다. 예를 들면 아래와 같습니다.

// 출력 결과를 묻는 문제

LocalDate date = LocalDate.now();
date.plusDays(5);
System.out.println(date);

역시 원본 date는 바뀌지 않으므로 그대로 출력됩니다. 다만 시간과 날짜 포맷 문제는 따로 외워 둘 필요가 있었습니다. java.sql.Date로 변환하는 문제는 보지 못했습니다.

기타

문제집에서는 복수 선택 문제인데 하나만 고르는 실수를 했던 적이 있습니다. 실제 시험에서는 단일 선택이면 라디오 버튼, 복수 선택이면 체크박스로 구분되어 있어서 그 부분은 걱정하지 않아도 됩니다. 체크박스 문제에서 하나만 고르면 다음 문제로 넘어가기 전에 경고도 나옵니다.

일부 문제는 루프 계산을 직접 해야 하지만, 대부분은 오래 걸리지 않고 바로 답할 수 있습니다. 시간 부족은 거의 없을 테니, 묻는 API의 특징만 잘 기억하면 합격은 어렵지 않다고 생각합니다.

반대로 저처럼 실무에 익숙해져 기본을 잊어버린 사람은 의외로 걸리는 부분이 있습니다. 하지만 어쨌든 시간을 끌 문제는 아니니, 코드가 주어지는 문제는 차분히 읽는 게 중요합니다. 정답은 사실상 세 가지뿐입니다. 결과 출력, 컴파일 에러, 실행 시 예외입니다. 먼저 이 셋 중 어디에 해당하는지부터 판단하면 훨씬 수월합니다.

정리하면, 실무 경험이 있다고 해서 Silver가 아주 쉬운 시험은 아니었습니다. 다만 묻는 포인트가 어느 정도 정해져 있으니, 헷갈리는 기본 문법과 API를 차분히 정리해 두면 충분히 대비할 수 있는 시험이라고 느꼈습니다.
