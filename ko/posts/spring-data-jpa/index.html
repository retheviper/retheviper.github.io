MyBatis 대신 JPA를 써 보기

Jul 10, 2020
9 minute read
日本語

개인적으로는 쿼리나 DB 플러그인처럼 DB 자체에서 데이터를 가공하는 방식은 그다지 선호하지 않습니다. 어떤 “처리"가 들어간다면 SQL은 가능한 최소한의 CRUD에만 두고, 나머지 작업은 애플리케이션에서 처리하는 편이 더 나은 설계라고 생각합니다. 시스템 전체 성능, 앱의 DB 독립성, 처리의 유연성 측면에서 더 낫다고 보기 때문입니다.

먼저 성능부터 보면, DB 서버와 AP 서버가 같은 수준의 머신을 쓰고 있다면 최적화된 쿼리만으로도 꽤 좋은 성능을 낼 수 있습니다. 하지만 데이터와 요청이 늘어나면 언제나 그렇게 대응할 수는 없습니다. 웹 애플리케이션에서 사용자가 늘어나면 보통 스케일 아웃1과 스케일 업2을 고민하게 됩니다. 그런데 DB 의존도가 높은 앱은 스케일 아웃 시 문제가 생길 수 있습니다. AP 서버는 여러 대로 늘려도 동작이 거의 변하지 않지만, DB 서버를 여러 대로 늘리면 데이터 정합성이라는 새로운 문제가 생깁니다. 모든 DB에 같은 데이터가 저장된다고 보장하고 싶다면, DB보다 AP 서버를 늘리는 편이 더 자연스럽습니다. 이런 이유로 데이터 처리는 가능한 한 애플리케이션 쪽에서 맡기는 편이 좋다고 생각합니다.

DB 독립성도 중요합니다. 요즘은 비용이 큰 Oracle에서 무료인 PostgreSQL이나 MySQL로 옮기는 시스템도 많습니다. 그런데 DB 쿼리나 플러그인에 많이 의존할수록 마이그레이션은 어려워질 수 있습니다. 미묘하게 다른 컬럼 타입부터 호환되지 않는 쿼리와 기능까지 모두 다시 검토해야 하니까요. 그래서 SQL은 가능한 한 단순하게 쓰는 편이 이식성에 유리합니다.

마지막으로 처리의 유연성입니다. RDBMS는 본질적으로 데이터를 효율적으로 저장하는 데 초점이 맞춰져 있습니다. 정규화된 테이블 구조는 데이터를 가공하기에는 그리 편하지 않습니다. 그래서 애플리케이션에서는 필요한 형태로 엔티티를 만들어 쓰지만, 그 엔티티에 맞게 SQL을 쓰려면 복잡한 Join이나 Where를 계속 신경 써야 합니다. 엔티티 안에 또 다른 엔티티가 들어가면 Join 여부까지 고민해야 합니다.

이런 이유로 저는 가능한 한 복잡한 쿼리를 줄이고, 애플리케이션 중심으로 개발할 방법이 없을까 생각했습니다. 처음에는 JDBC를 다루다가 MyBatis를 썼지만, MyBatis도 경우에 따라 쿼리를 직접 써야 하는 것은 마찬가지였습니다. XML이 필요한 경우도 있었고요. 최근에는 원격근무 덕분에 출퇴근 시간이 줄어들어 퇴근 후에는 공부나 자작 앱 구현을 하고 있는데, 그 과정에서 더 단순하고 애플리케이션 친화적인 DB API가 없을까 찾다가 JPA를 알게 됐습니다.

JPA란

JPA는 Java Persistence API의 약자로, Java가 제공하는 표준 API입니다. 다만 표준 API라고 해도 인터페이스만 정의된 것이므로, 이를 구현한 실제 구현체가 필요합니다. JPA 구현체로 유명한 것이 Hibernate이고, 이를 Spring에서 더 쓰기 쉽게 감싼 것이 Spring Data JPA입니다.

서론이 길었지만, 결국 제가 말하고 싶은 것은 Spring Data JPA를 쓰면 개발 방식 자체가 꽤 달라진다는 점입니다. SQL을 중심에 두고 생각하기보다, 엔티티와 관계를 중심으로 애플리케이션을 구성하게 됩니다. 저처럼 SQL 중심 개발에 답답함을 느끼던 사람에게는 이 점이 꽤 크게 다가왔습니다. 단순한 쿼리는 자동으로 만들어 주고, Join이나 참조 관계도 엔티티 기준으로 다룰 수 있어서 직접 SQL을 많이 쓰지 않아도 되기 때문입니다.

그럼 JPA, 정확히는 Spring Data JPA의 특징과 사용법을 하나씩 보겠습니다.

JPA의 특징

먼저 JPA의 특징입니다. 엄밀히는 “장점"이라고 보는 편이 맞을 수도 있습니다.

자동 그래프 탐색

다음과 같은 클래스가 있다고 해 봅시다.

class Team {
    private long id;
    private String name;
}

그리고 팀에는 소속 선수들이 있습니다.

class Player {
    private long id;
    private String name;
    private Team team;
}

기존 SQL로 이 선수 객체를 조회하려면 보통 아래 순서가 필요합니다.

  1. Team 테이블 조회
  2. 조회한 데이터를 Team 객체에 넣기
  3. Player 테이블 조회
  4. 조회한 데이터를 Player 객체에 넣기
  5. Player 객체에 Team 객체 연결

조인이 많아질수록 이 절차도 늘어나고, SQL도 더 길어집니다. 예를 들어 Team 클래스 안에 Region이 있고, Region 안에 Country가 들어 있다면 각 테이블을 조회하고 조인하는 쿼리를 따로 작성해야 하고, 객체도 전부 매핑해야 합니다. 중간 단계가 하나라도 빠지면 코드가 예상대로 동작하지 않습니다.

JPA는 이런 참조 관계를 자동으로 처리해 줍니다. 예를 들면 아무것도 하지 않아도 아래처럼 쓸 수 있습니다.

// Repository에서 Player 객체를 가져온다
Player player = repository.findById(1);
// Player가 속한 Team을 가져온다
Team team = player.getTeam();

성능 최적화

자동으로 그래프 탐색을 해 준다면, Player를 조회할 때마다 Team까지 같이 조회하는 쓸데없는 작업이 생기는 건 아닌지 걱정될 수 있습니다. 하지만 실제로는 Lazy Loading 덕분에 getTeam()을 호출하는 시점에 다시 Select를 날립니다. 그래서 Player 정보만 필요하면 그것만 조회합니다.

경우에 따라서는 쿼리가 여러 번 발생할 수 있다는 의문도 있습니다. JPA는 캐시, 배치, Fetch Type 지정으로 이 문제를 해결합니다.

  • 캐시: 같은 엔티티이고 같은 트랜잭션이라면 두 번째 이후 Select는 캐시에서 가져옵니다
  • 배치 처리: 여러 쿼리를 한 번에 전달하는 방식을 제공합니다
  • Fetch Type: Lazy Loading을 쓸지, 처음부터 Join해서 가져올지 어노테이션으로 선택할 수 있습니다

자동 쿼리 생성

앞에서 Repository로 객체를 조회하는 예를 봤듯이, JPA가 자동으로 만들어 주는 쿼리가 있기 때문에 이런 방식이 가능합니다. JPA가 제공하는 어노테이션과 인터페이스를 쓰면 자동 생성된 쿼리를 활용할 수 있습니다.

즉, 엔티티로 쓸 클래스와 JPA의 어노테이션, 인터페이스를 연결해 두면 매번 쿼리를 직접 쓰고 객체에 매핑할 필요가 없어집니다. 물론 직접 쿼리를 써도 되므로, 자동 생성 쿼리를 쓸지 말지는 선택할 수 있습니다.

이 자동 생성은 CRUD에만 한정되지 않습니다. 엔티티가 바뀌어 필드를 추가해야 할 때 DDL도 자동 생성할 수 있습니다. 이때 Drop-Create로 할지 Alter로 할지는 애플리케이션 프로퍼티의 Hibernate 설정으로 조정할 수 있습니다.

JPA 사용법

이제 실제로 JPA를 코드에서 어떻게 쓰는지 보겠습니다.

Entity

JPA에서는 엔티티와 그 필드, 즉 테이블의 컬럼과 참조 관계를 어노테이션으로 쉽게 정의할 수 있습니다. 예를 들어 이런 식입니다.

// Entity로 지정한다
@Entity
class Car {

    // 자동 증가 PK
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private long id;

    // Unique이면서 NonNull
    @Column(nullable = false, unique = true)
    private String name;

    // Company와는 다대일 관계
    @ManyToOne
    @JoinColumn(name = "company_id")
    private Company company;

    // Insurance와는 일대다 관계, Lazy Loading 사용
    @OneToMany(fetch = FetchType.LAZY)
    @JoinColumn(name = "insurances_id")
    private List<Insurance> insurances;
}

이렇게 엔티티를 정의해 두면, Select한 시점부터 이 클래스와 연결된 Company나 Insurance 같은 관계도 참조할 수 있습니다.

Repository

JPA는 여러 Repository 인터페이스를 제공합니다. 자주 쓰는 것은 JpaRepository, CrudRepository, PagingAndSortingRepository 등입니다. 각각 기본 메서드 구성이 조금씩 다릅니다. 예를 들어 CrudRepository는 CRUD용 메서드, PagingAndSortingRepository는 페이징용 메서드를 제공합니다.

// 레코드 수를 센다
long count = repository.count();

// 모든 레코드를 가져온다
Iterable<Car> cars = repository.findAll();

// 레코드 하나를 가져온다
Optional<Car> car1 = repository.findById(1);

// 레코드 하나를 삭제한다
repository.delete(car2);

// 레코드를 저장한다(Upsert)
Car saved = repository.save(car3);

엔티티를 정의하고, 그에 맞는 JPA Repository를 extend해 두면 그 Repository로 쿼리를 보낼 수 있습니다. 기본 메서드만으로 부족하면, 이름 규칙에 따라 쿼리 메서드를 직접 정의해 자동으로 쿼리를 만들 수도 있고, @Query로 직접 쿼리를 적을 수도 있습니다.

public interface CarRepository extends CrudRepository<Car, Long> {

    // name으로 조회하는 커스텀 쿼리 메서드
    boolean existsByName(String name);

    // 어노테이션으로 쿼리를 지정하는 경우
    @Query("select c from Car c where c.name = ?1")
    Optional<Car> findByName(String name);
}

그 외

2018년에는 Spring Data JDBC도 발표됐습니다. Spring Data JPA는 JPA + Hibernate 계층을 거치는 반면, Spring Data JDBC는 JDBC를 직접 다루므로 성능 면에서 유리할 수 있습니다. 또한 구현이 단순해서 JPA가 가진 몇몇 장점을 걷어내기도 합니다. 예를 들면 캐시나 Lazy Loading이 없습니다.

다만 Spring Data JDBC는 출시된 지 오래되지 않아서 쿼리 메서드 자동 생성 같은 기능이 부족하고, 테이블이나 컬럼 명명 규칙도 JPA와 다릅니다. JPA에서 마이그레이션하려면 아직 고려할 점이 많습니다. 느낌상 Node.js와 Deno의 관계와 비슷합니다.

Spring Data JDBC는 저도 아직 깊게 알지는 못하고 샘플 코드를 조금 테스트해 본 정도입니다. 그래도 이미 Spring Data JPA를 써 본 사람이라면 금방 적응할 수 있을 것 같은 구조였습니다. 패키지가 달라도 어노테이션과 인터페이스 사용 방식이 아주 낯설지는 않았고, 체감상 가장 큰 차이는 @Entity 같은 어노테이션이 필요 없다는 점 정도였습니다. 개인 프로젝트에서는 한 번 시도해 볼 만하지만, 엔터프라이즈 수준에서는 아직 더 검증이 필요하다고 봅니다.

또 Spring 5에서 Reactive 프로그래밍을 지원하는 WebFlux가 소개되면서, 마찬가지로 non-blocking에 대응하는 Spring R2DBC도 주목받고 있습니다. 기존 Spring 프로젝트에 WebFlux를 굳이 지금 당장 넣을 필요는 없어 보이듯, R2DBC도 지금 바로 도입할 필요는 없을 수 있습니다. 하지만 non-blocking으로 성능을 개선할 수 있는 경우3는 분명 있으니, 상황에 따라 도입을 검토할 가치는 있다고 생각합니다.

마지막으로

저는 Spring으로 애플리케이션을 만들 때 가능한 한 XML을 배제하고 Java 코드만으로 관리하고 싶었습니다. 그래서 쿼리를 자동 생성해 준다는 점이 특히 매력적으로 느껴졌습니다. 실제로 써 보면 사용법도 생각보다 단순하고, 테이블 구조가 조금 바뀌더라도 수정 흐름이 비교적 자연스럽습니다.

점유율 면에서도 사실상 표준처럼 쓰이는 경우가 많아서, JPA는 다른 ORM보다 자주 선택됩니다. MyBatis만 써 봤다면 한 번 직접 만져 보는 것만으로도 개발 감각이 꽤 달라질 수 있습니다.

JPA 외에도 Spring Data 계열에는 여러 라이브러리가 있고, 각각 지향하는 방향도 조금씩 다릅니다. 아직 하나를 겨우 만져 본 정도이긴 하지만, 라이브러리가 왜 그렇게 설계됐는지 살펴보는 것만으로도 얻는 게 많았습니다. 적어도 MyBatis만 쓰던 때보다 선택지가 훨씬 넓어졌다는 점은 분명합니다.
