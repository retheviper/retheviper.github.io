Kotlin

Kotlin의 앞으로를 이야기해 본다

May 22, 2022
18 minute read

1년 정도 서버 사이드 Kotlin을 다루다 보니 문득 “지금의 Kotlin은 어디까지 와 있고, 앞으로는 어떻게 될까?“라는 생각이 들었습니다. 관점은 여러 가지가 있겠지만, 일단 시장에서의 수요와 전망, 그리고 Kotlin이라는 언어의 현재 위치를 스스로 어떻게 보고 있는지 정리해 보고 싶었습니다.

요즘은 한 언어만 깊게 파는 개발자보다 여러 언어를 다룰 수 있는, 이른바 Polyglot 개발자가 더 많이 요구된다는 이야기를 자주 듣습니다. 실제로 저 역시 프로젝트에 따라 다양한 언어를 접하는 경우를 많이 봤습니다. 게다가 지금은 인터넷에 문서와 자료가 넘치고, Udemy나 Coursera처럼 양질의 강의를 제공하는 서비스도 많아서 처음 접하는 언어라도 입문 자체가 아주 어렵지는 않은 시대가 되었습니다. 그런 점에서 보면, 내가 쓰는 언어가 주류인지 아닌지는 예전만큼 절대적인 문제는 아닐지도 모릅니다.

그래도 상황에 따라서는 한 언어에 집중하는 편이 더 중요할 때가 있습니다. 예를 들어 학생이나 주니어 개발자, 혹은 비전공에서 개발자로 전환하려는 사람에게 처음부터 두세 가지 언어를 동시에 익히라고 요구하는 것은 현실적으로 쉽지 않습니다. 프런트엔드 엔지니어가 당장 쓸 일도 없는 Go나 Java 같은 백엔드 언어를 우선적으로 공부해야 할 이유도 없습니다. 회사 입장에서도 여러 언어를 다룰 수 있는 인재만 찾는다면 채용 기준이 지나치게 높아질 수 있습니다. 그래서 저는 여전히 시장에서 “언어의 위상"이라는 요소를 무시할 수 없다고 생각합니다.

이번 글은 그런 관점에서, 다소 주관적이긴 하지만 다른 언어 및 분야와 비교해 가며 Kotlin의 앞으로를 어떻게 보고 있는지 정리해 보려 합니다.

vs 자바

Better Java라는 시선

Kotlin/JVM을 Java와 비교할 때 흔히 나오는 평가는 “Java와 완벽하게 상호 운용되고, 성능도 크게 다르지 않다"는 것입니다. 여기에 확장 함수, 코루틴, 스코프 함수, 널 안정성처럼 현대적인 기능까지 갖추고 있으니, 겉으로만 보면 better java라고 불려도 이상하지 않습니다. Java 쪽에서 JVM이 개선되면 그 이점이 Kotlin에도 어느 정도 이어진다는 점도 분명 장점입니다. Java 역시 1.8 이후 릴리스 주기가 빨라졌지만, 애플리케이션 개발자 관점에서는 아직 Kotlin이 더 편리하게 느껴지는 부분이 있습니다.

이렇게만 보면 Kotlin이 Java를 완전히 대체할 수 있는 언어처럼 들립니다. 하지만 실제 산업은 그렇게 단순하게 움직이지 않습니다.

우선 Java는 오랫동안 세계에서 가장 널리 쓰인 언어 가운데 하나였고, 지금도 여러 조사에서 꾸준히 상위권을 유지합니다1. 이는 단순히 “앞으로도 많이 쓰일 것"이라는 뜻만이 아니라, 지금까지 Java로 구축된 시스템과 애플리케이션이 압도적으로 많다는 뜻이기도 합니다. 이미 Java 기반으로 돌아가는 서비스가 많은 이상, 특별한 계기가 없는 한 유지보수와 확장 개발을 위해 Java 엔지니어에 대한 수요는 계속 남을 수밖에 없습니다.

또 한 가지는, Java의 장점을 활용하면서 더 현대적인 개발 경험을 제공하겠다는 목표로 등장한 JVM 언어가 Kotlin만은 아니라는 점입니다. Clojure, Scala, Groovy 등도 비슷한 포지션에서 각자의 영역을 만들어 왔지만, 그렇다고 해서 “Java를 넘어섰다"는 평가를 받지는 못했습니다. Kotlin이 그들보다 더 넓게 받아들여지고 있는 것은 맞지만, “JVM 언어다”, “Java보다 모던하다"는 이유만으로 Java를 대체할 것이라고 단정하기는 어렵습니다.

모바일에서는 Android 진영에서 Java보다 Kotlin을 채택하는 사례가 많이 늘었습니다. 다만 여기에는 Oracle과 Google의 소송 이슈로 Android 쪽에서 Java를 오래 1.8 수준에 묶어 두었던 사정도 있었습니다. 반면 웹 백엔드에서 많이 쓰이는 OpenJDK는 그런 제약이 상대적으로 적었고, Java 17부터는 Oracle JDK도 무료로 사용할 수 있게 되면서 상황이 꽤 달라졌습니다. 그래서 모바일에서의 흐름을 그대로 웹으로 가져오기는 어렵다고 생각합니다.

물론 JetBrains도 이런 현실을 모르지 않았기 때문에 Kotlin을 처음부터 Java와의 상호 운용성을 강하게 의식한 언어로 설계했습니다. 결국 Kotlin의 전략은 “기존 Java 애플리케이션을 한 번에 갈아엎는다"기보다는, 부분적으로 도입하고 신규 개발에서 점유율을 늘려 가는 쪽에 가깝다고 봅니다. 이 전략 자체는 충분히 설득력 있고, 기업이 Java와 Kotlin 두 언어를 함께 운영하는 데 거부감만 없다면 실제 도입 장벽도 높지 않습니다. 저 역시 Java 프로젝트에서 Kotlin으로 넘어갈 때 큰 어려움은 없었습니다.

Kotlin도 점점 강해지고 있다

최근 프레임워크와 라이브러리 생태계를 보면, Kotlin은 여전히 모바일 바깥에서의 인지도는 Java보다 약하지만 조금씩 존재감을 넓혀 가고 있습니다. 예를 들어 제가 업무에서 쓰는 Spring Boot, Jackson, AWS SDK처럼 웹 개발에서 영향력이 큰 도구들이 Kotlin을 지원하고 있습니다. jOOQ, jooby, Javalin처럼 Java와 Kotlin을 함께 지원하는 라이브러리도 점점 늘고 있습니다.

기존 Java 라이브러리를 Kotlin 친화적으로 다듬은 사례도 있고, 처음부터 Kotlin을 중심에 두고 설계된 라이브러리도 적지 않습니다. Kotlin Serialization, Klaxon, DGS, Ktorm, Kotest, MockK, Exposed, KMongo, Xodus, Koin, Kodein-DI 같은 도구들이 그렇습니다. 몇 년 전만 해도 Kotlin이 Java 생태계에 기대어 가는 느낌이 강했다면, 지금은 Kotlin만으로도 웹 애플리케이션을 충분히 구축할 수 있는 수준까지 올라왔다고 느낍니다.

결국 아직은 Java가 규모와 인지도에서 훨씬 앞서 있지만, Kotlin도 이제는 충분히 경쟁할 만한 힘을 갖추고 있습니다. 앞으로 상황이 더 달라질 가능성은 분명히 있다고 봅니다.

vs Go

“빠르다"는 미덕

업무에서 Go를 써 본 입장에서, Kotlin과 비교했을 때 Go의 가장 큰 장점은 역시 “정말 빠르다"는 점입니다. 네이티브 언어라서 런타임 성능이 좋은 것도 있지만, 컴파일과 빌드, 테스트 피드백이 빠르다는 점이 특히 크게 느껴졌습니다. 코드를 수정한 뒤 단위 테스트를 반복해서 돌리는 과정에서 Go 프로젝트는 Kotlin 프로젝트보다 훨씬 가볍게 느껴졌습니다. 물론 Kotlin 쪽은 Spring을 쓰고 테스트 수도 더 많았으며, 빌드 방식 차이도 있었겠지만 체감상 차이는 분명했습니다.

또 GitHub 생태계의 패키지를 그대로 가져다 쓰기 쉽고, 별도 라이브러리 없이도 struct를 바로 JSON으로 다룰 수 있는 점, omitempty 같은 기능이 자연스럽게 녹아 있는 점도 인상적이었습니다. 생성되는 바이너리 크기가 작다는 점까지 포함하면 Go는 웹 백엔드, 특히 서버리스나 마이크로서비스 환경에 잘 맞는 언어라는 생각이 듭니다.

예전에는 서버가 VM 중심으로 옮겨가던 시기라서 JVM 언어의 부담이 하드웨어 성능 향상으로 어느 정도 가려졌습니다. 그런데 서버리스와 마이크로서비스가 확산되면서 JVM의 특성이 다시 비용과 운영 문제로 드러나고 있습니다. 서버리스에서는 JVM이 완전히 올라오는 시간과 콜드 스타트가 부담이 되고2, 마이크로서비스 환경에서는 힙 메모리와 스토리지 사용량이 결국 인스턴스 비용 증가로 이어질 수 있습니다.

Kotlin이 곧 느리다는 뜻은 아니다

그렇다고 Kotlin이 무조건 느리다고 보기는 어렵습니다. 서버리스 쪽에서는 Kotless 같은 시도가 있었고, GraalVM을 활용해 네이티브 빌드를 지원하는 Quarkus나 Spring Native 같은 흐름도 나왔습니다. 즉, JVM 언어 진영도 최근 트렌드에 맞춰 개선을 이어 가고 있습니다.

런타임 성능 역시 단순 비교로 결론 내리기 어렵습니다. JIT 최적화 덕분에 JVM 계열 언어가 Go에 밀리지 않는 경우도 있고, 벤치마크를 봐도 Kotlin/JVM 비교, Kotlin/Native 사례처럼 어떤 작업에서는 Go가, 또 어떤 작업에서는 Kotlin이 더 나은 결과를 보입니다.

게다가 Go 역시 기능이 확장되면서 트레이드오프가 생길 수 있습니다. 예를 들어 Go 1.18에서 제네릭이 도입된 뒤, 경우에 따라 성능이나 단순성에 영향을 줄 수 있다는 논의도 있었습니다. 결국 언어가 발전할수록 속도만으로 모든 장점을 유지하기는 어렵습니다.

그래서 Kotlin과 Go를 비교할 때, 성능만 보고 Go를 고집해야 한다고 보지는 않습니다. 다만 실제 언어 선택은 성능 외에도 생산성, 운영 환경, 클라우드 지원, 인력 수급 같은 요소들이 함께 작용합니다. 제가 이직을 고민하던 시기에도 서버 사이드 채용은 Kotlin보다 Go가 훨씬 많았습니다. 단순히 성능만이 기준이었다면 이런 시장 구도가 설명되지는 않을 것입니다. Google이 밀고 있는 언어라는 점, 생산성과 배포 편의성, 생태계 확장성 같은 요소가 함께 작용한 결과라고 보는 편이 맞겠습니다.

그래도 당분간은 Go가 유리하다

또 하나는 인지도 문제입니다. Kotlin은 네이티브 이미지까지 지원하고 성능도 충분히 경쟁력이 있지만, 여전히 많은 사람들에게는 “Android용 언어"라는 인상이 강합니다. 이 인식이 엔지니어와 기업 양쪽에서 바뀌지 않는 한, 서버 사이드 시장에서는 당분간 Go가 더 유리한 위치를 유지할 가능성이 높습니다.

Go는 배우기 쉽고 문법이 단순하다는 점도 큰 장점입니다. 제게는 Kotlin의 표현력이 더 간결하고 편하게 느껴지지만, 반대로 Go 쪽이 훨씬 단순해서 집중하기 좋다고 느끼는 사람도 많습니다. 키워드 수가 적고 언어 규칙이 비교적 단순하다면, 로직 자체에 집중하기 쉬운 것도 사실입니다. CLI 도구나 작은 서비스에서 Go가 많이 쓰이는 것도 그런 특성과 관련이 있을 겁니다. 저 개인적으로는 간단한 도구는 Python으로 만드는 편이지만, 가볍게 무언가를 만들 때 Kotlin보다 Go가 더 부담이 적다고 느끼는 사람은 충분히 많을 것 같습니다. 그런 흐름이 다시 Go의 인기로 이어지는 것도 자연스러운 일입니다.

vs Rust

성능 우위

Rust는 GC 없이도 네이티브 성능을 내는 언어라서, Go보다 더 높은 성능을 기대할 수 있는 경우가 많습니다. 다만 Kotlin과 비슷하게, Rust 역시 대중적 인지도 측면에서는 아직 한계가 있다고 느낍니다. 원래부터 C/C++을 대체하는 목적이 강했던 언어라 그런지, 많은 사람에게는 여전히 임베디드나 시스템 프로그래밍 쪽 이미지가 강합니다.

하지만 실제 채택 사례를 보면 Figma, 1Password, Discord, Dropbox, Mozilla, LINE, npm, Cloudflare처럼 꽤 다양한 조직이 Rust를 활용하고 있습니다. CLI 도구나 웹 프레임워크도 꾸준히 늘고 있어 앞으로 더 기대되는 언어라는 생각은 분명 듭니다. 다만 인기가 더 높아지더라도, Rust는 웹 애플리케이션 전반보다는 시스템 프로그래밍이나 성능이 중요한 영역 쪽으로 계속 강점을 유지할 가능성이 더 커 보입니다.

Kotlin을 네이티브로 가져가면

Kotlin과 Rust를 직접 비교할 때는 자연스럽게 Kotlin/Native가 떠오릅니다. 이론적으로는 둘 다 네이티브 환경을 노릴 수 있지만, 실제 시장에서의 포지션은 꽤 다릅니다. Rust는 임베디드와 시스템 프로그래밍에서 C/C++ 대체 언어로 점점 자리를 잡아 가는 반면, Kotlin/Native는 “어디에서 결정적으로 강한가"가 상대적으로 모호합니다.

특히 Kotlin/Native가 LLVM 기반이라는 점은 흥미롭지만, 이미 GraalVM을 이용한 네이티브 이미지와 네이티브 대응 웹 프레임워크가 등장한 지금은 그 포지션이 더 애매하게 느껴지기도 합니다. Objective-C나 C/C++와의 상호 운용성이 장점으로 거론되긴 하지만, 그런 요구가 강한 환경이라면 애초에 해당 언어들을 직접 쓰는 편이 더 자연스러운 경우도 많습니다. 물론 Rust는 소유권 개념 때문에 난도가 높고, 그에 비해 Kotlin/Native는 상대적으로 작성 부담이 적을 수 있습니다. 하지만 네이티브를 택하는 이유가 대체로 성능과 제어력에 있다면, GC가 있는 Kotlin은 불리하게 보일 수밖에 없습니다.

결국 Kotlin/JVM과 Rust는 강점을 발휘하는 영역이 꽤 다르기 때문에, 큰 변화가 없는 한 서로의 시장을 정면으로 잠식하기보다는 각자 다른 방향으로 성장할 가능성이 높다고 봅니다. 직접적인 경쟁은 오히려 Kotlin/Native와 Rust 사이에서 일부 생길 수 있겠지만, “정말 네이티브가 중요한 곳"에서는 Rust가 더 자주 선택될 가능성이 높아 보입니다.

vs Python

만능 도구

최근 몇 년 동안 가장 큰 인기를 끈 언어 중 하나가 Python인데, Kotlin과 비교하면 성격 차이가 꽤 분명합니다. 제 경우 일상적인 자동화나 간단한 도구를 만들 때는 Python을 자주 쓰고, 본격적인 웹 애플리케이션을 만들 때는 Kotlin을 더 선호합니다. 물론 Python은 범용 언어이기 때문에 대규모 애플리케이션 개발에 적합하지 않다는 뜻은 아닙니다. 실제로 Uber, Google, PayPal, Netflix 같은 기업도 Python을 사용하고 있고, Instagram의 서버 사이드가 Python 기반이라는 이야기도 널리 알려져 있습니다.

다만 개인적으로는 Python이 데이터 사이언스나 AI처럼 강한 분야에서는 확실한 존재감을 갖고 있지만, 대규모 서비스 개발에서는 한계가 비교적 분명하다고 느낍니다. 제가 대형 Python 서비스를 직접 운영해 본 것은 아니어서 어디까지나 인상에 가깝지만, 서버 사이드에서 Python을 도입한 회사는 스타트업 비중이 높은 편이고, 서비스가 커질수록 인터프리터 언어 특유의 유지보수 부담이 드러날 가능성이 있습니다. 타입 힌트가 있긴 하지만 TypeScript처럼 컴파일 타임 검증이 강제되는 구조는 아니고, GIL 같은 제약도 여전히 존재합니다. 그래서 프로토타입은 Python으로 빠르게 만들고, 이후 다른 언어로 옮기는 사례가 나오는 것도 자연스러운 흐름처럼 보입니다.

Python만의 영역도 아니다

반대로 Kotlin도 데이터 사이언스 용도로 활용할 수 있는 가능성을 갖고 있습니다. Jupyter 지원도 있고, JetBrains가 강조하는 것처럼 정적 타입, 널 안정성, 성능 측면에서 분명 강점이 있습니다. 문제는 이미 Python이 시장을 사실상 장악한 영역에서 Kotlin이 얼마나 점유율을 넓힐 수 있느냐입니다. 아무리 기술적으로 장점이 있어도, 실제 사용자와 생태계가 따라오지 않으면 확장에는 한계가 있습니다.

Python은 입문 장벽이 낮고 교육 자료도 풍부해서 비개발자까지 넓게 유입되는 언어입니다. 반면 Kotlin은 아직 그런 저변 확대 면에서는 약한 편입니다. 그래서 앞으로도 데이터 사이언스처럼 Python이 본래 강한 분야에서는 큰 흐름이 쉽게 바뀌지 않을 것 같습니다. 웹 쪽에서는 Kotlin이 더 안정적이고 구조적인 개발 경험을 줄 수 있지만, 실제 선택지는 Kotlin과 Python만 있는 것이 아니기 때문에, 결국 두 언어만 놓고 비교했을 때의 이야기라는 점은 감안해야 합니다.

vs 자바스크립트

다재다능함의 끝판왕

한 언어로 거의 모든 것을 할 수 있는 언어를 꼽으라면, 예전에는 Java였고 그다음에는 Python이었으며, 지금은 JavaScript라고 해도 과장이 아닐 겁니다. 프런트엔드, 백엔드, 모바일, 데이터 영역까지 폭넓게 쓰이고 있고, 런타임 성능 문제도 Deno 같은 새로운 런타임이나 V8 엔진의 꾸준한 개선으로 계속 보완되고 있습니다. 정적 타입 문제도 TypeScript의 확산으로 상당 부분 해결되었습니다. 이런 면만 보면 정말 빈틈없는 언어처럼 보입니다.

특히 프런트엔드에서는 JavaScript를 빼고는 논하기 어려운 현실이 있습니다3. WebAssembly 같은 기술도 성장하고 있지만, 이 역시 JavaScript를 완전히 대체하기보다는 다른 방향으로 웹 생태계를 확장하는 느낌에 가깝습니다. 설령 특정 분야에서 JavaScript의 위상이 달라지더라도, 언어 자체가 사라질 가능성은 매우 낮다고 생각합니다. 그런 의미에서 Kotlin이 이 영역으로 깊숙이 들어가는 일은 생각보다 훨씬 어렵습니다.

Kotlin으로 프런트엔드를 할 수 있을까

Kotlin 쪽에서도 가만히 있는 것은 아닙니다. Kotlin/JVM, Kotlin/Native와 별개로 Kotlin/JS가 있고, JetBrains도 여기에 꾸준히 힘을 싣고 있습니다. 여기에 Compose Multiplatform까지 더해지면서 모바일뿐 아니라 웹과 데스크톱 앱까지 Kotlin으로 다루려는 시도도 계속 이어지고 있습니다. 개인적으로는 사이드 프로젝트를 가능하면 Kotlin 하나로 통일하고 싶기 때문에 이런 흐름을 꽤 응원하는 편입니다.

다만 현실적으로 보면 아직 모바일 밖에서는 대중적인 선택지라고 보기 어렵습니다. 생태계가 아직 얇고, 라이브러리 부족이나 잦은 변화처럼 신기술 특유의 불안정성도 있습니다. 무엇보다 “왜 굳이 여기서 Kotlin을 써야 하는가?“에 대한 답이 개인이나 기업 모두에게 아직 충분히 강하지 않은 경우가 많습니다.

백엔드에서는 Kotlin이 Java의 대안으로 검토될 여지가 여전히 큽니다. 특히 기존에 Java를 쓰던 분야라면 JVM의 안정성, 검증된 계산 환경, 운영 경험 같은 자산이 있기 때문에, 언어를 바꿔야 한다면 JavaScript보다 Kotlin을 먼저 고려할 가능성이 높다고 봅니다. 반대로 스타트업처럼 기술 스택 수를 줄이는 것이 중요한 경우나, 프로토타입을 빠르게 만드는 것이 핵심인 경우라면 JavaScript 쪽이 더 강할 수 있습니다. 결국 프런트엔드부터 백엔드, 모바일, 데스크톱까지 한 언어로 밀고 가고 싶다면 JavaScript 진영이 여전히 가장 강력한 선택지이고, 그런 상황에서는 Kotlin이 선택되지 않을 가능성이 높습니다.

vs Dart

GUI 영역에서의 강점

Dart는 언어 그 자체보다 Flutter의 성장으로 더 많이 주목받고 있습니다. 처음에는 모바일 크로스플랫폼 개발 도구로 화제가 되었지만, 지금은 사실상 Dart의 존재감 자체가 Flutter와 함께 커졌다고 봐도 무방합니다. 이 영역에서 Flutter의 대표적인 경쟁자는 React Native일 텐데, 최근 몇 년 흐름만 보면 적어도 체감상 Flutter 쪽 존재감이 더 강해진 느낌도 있습니다.

물론 현실은 그렇게 단순하지 않습니다. 예를 들어 프런트엔드 엔지니어가 모바일까지 함께 맡고 있고, 이미 웹 프런트엔드에서 React를 쓰고 있다면 여기서 Flutter를 새로 도입할 유인은 크지 않을 수 있습니다. 결국 어떤 프레임워크가 더 좋아 보이느냐와 실제 조직이 무엇을 선택하느냐는 다른 문제입니다.

제가 느끼는 Dart의 가장 큰 약점은, 원래의 목표가 어땠든 지금도 언어 자체의 개성이 상대적으로 약하다는 점입니다. 조금 써 보면 C 계열 언어로서 낯설지는 않지만, “이 점 때문에 꼭 Dart를 쓰고 싶다"는 강한 매력이 언어 차원에서는 크게 와닿지 않았습니다. 지금은 Flutter 덕분에 사용률이 올라가고 있지만, Flutter 밖의 영역까지 얼마나 확장될 수 있을지는 여전히 의문이 남습니다.

다른 가능성도 있긴 하다

그럼에도 Dart의 가능성을 완전히 좁게 볼 필요는 없습니다. 예전부터 Google의 차세대 OS로 거론되는 Fuchsia에서 Dart가 핵심 개발 환경이 될 수 있다는 이야기가 있었고, 실제로 어떤 방향으로 굳어질지는 아직 불확실합니다. 만약 정말 Android의 다음 세대 플랫폼 같은 위치를 차지하게 된다면, Dart의 위상은 지금과 비교할 수 없을 정도로 달라질 수도 있습니다.

또 Dart 역시 프로그래밍 언어이기 때문에, 앞으로 프레임워크와 라이브러리가 어떻게 발전하느냐에 따라 충분히 다른 그림이 나올 수 있습니다. 이 리포지토리를 보면 서버 사이드 프레임워크도 이미 몇 가지 존재합니다. “하나의 언어로 가능한 한 많은 것을 해결하고 싶다"는 관점에서는 오히려 Dart가 Kotlin보다 유리해 보이는 면도 있습니다. Kotlin 쪽의 Kotlin/Multiplatform Mobile은 기본적으로 비즈니스 로직 공유에 초점을 맞추고 있어서, 결국 iOS 코드는 따로 써야 하기 때문입니다.

물론 일부 기업처럼 “UI는 Flutter, 비즈니스 로직은 네이티브"라는 전략도 가능하겠지만, 아직 대중적인 방식이라고 하기는 어렵습니다. Swift도 Vapor 같은 서버 프레임워크를 통해 서버 사이드 가능성을 보여 주지만, 실제 채택 사례가 많지 않은 것을 보면 “기술적으로 가능하다"는 사실만으로는 충분하지 않다는 점을 다시 느끼게 됩니다.

모바일에서도 더 강해진다

특히 2022년 Google I/O에서 보였듯이, Flutter 3는 성능 개선과 Flutter Desktop 정식 릴리스 등 여러 면에서 눈에 띄는 발전을 보여 주었습니다. Flutter를 채택하는 기업도 늘고 있기 때문에, 이런 발전이 실제 시장에 반영되는 것도 시간문제라고 생각합니다. 물론 네이티브 앱 개발 수요도 계속 남겠지만, 크로스플랫폼만으로 충분한 분야가 더 많아진다면 어느 쪽이 주류가 될지는 꽤 분명해질 수 있습니다.

이런 흐름은 Kotlin 입장에서는 꽤 민감한 문제입니다. 지금도 모바일은 Kotlin의 가장 강한 홈그라운드라고 할 수 있는데, 바로 그 영역에서 Flutter가 빠르게 성장하고 있기 때문입니다. 그래서 앞으로는 Kotlin만의 강점을 더 또렷하게 만들어 갈 필요가 있어 보입니다. 앞서 언급한 Kotlin/Multiplatform Mobile 같은 시도가 그런 역할을 어느 정도 해 줄 수 있기를 기대하고 있습니다. 그 밖에도 Kotlin은 모바일 외에 활용할 수 있는 분야가 많으니, 영역 간 연결성을 더 강화한다면 앞으로도 충분한 경쟁력을 가질 수 있을 것입니다.

마지막으로

이번 글은 평소보다 제 생각이 훨씬 많이 들어간 글이라 다소 편향된 판단이 있을 수도 있습니다. 그래도 Kotlin을 쓰는 개발자로서 지금 시점의 인상을 한 번 정리해 보고 싶었습니다. 물론 모바일, 프런트엔드, 데이터 사이언스, DevOps 등 각 분야에서 더 깊이 일하는 분들이 보기에는 부정확하거나 지나치게 단순화한 부분도 있을 수 있습니다.

그래도 한 명의 엔지니어로서, 그저 흐름을 수동적으로 지켜보기만 하기보다는 내가 쓰는 기술과 관심 있는 분야를 기준으로 나름의 판단을 정리해 보는 일도 의미가 있다고 생각합니다. 이런 글을 남겨 두면 나중에 실제 시장이 어떻게 바뀌었는지, 제 전망이 얼마나 맞았는지도 다시 돌아볼 수 있을 테니까요.

정보를 촘촘히 정리한 글이라기보다는 생각을 풀어낸 글에 가깝지만, 이 글이 Kotlin을 다시 한 번 떠올리는 계기가 되었다면 그걸로 충분합니다.
