최근에 [FileTransporter](https://github.com/retheviper/FileTransporter)를 크게 손봤습니다. 원래는 개인 용도로 단순한 파일 브라우저 서버처럼 쓰려고 만든 앱이라, 핵심 기능만 어느 정도 구현해 둔 뒤 오랫동안 거의 방치해 두고 있던 상태였습니다.

사실 이런 식의 프로젝트가 꽤 많습니다. 해 보고 싶은 것은 많은데, 막상 다시 손대려고 하면 환경 구축, 아키텍처 점검, 에러 수정, 관련 조사 같은 것에서 먼저 막히고, 그러다 보면 다시 우선순위에서 밀려 버립니다. 이번에도 본질적으로는 그런 종류의 숙제가 쌓여 있던 셈이었습니다.

그런데 요즘은 코딩 에이전트가 꽤 실용적인 수준까지 올라와서, 기존 코드베이스를 실제로 수정하는 작업 자체가 예전보다 훨씬 가벼워졌습니다. 특히 이번에는 단순히 코드 몇 줄을 고치는 수준이 아니라, `AGENTS.md` 같은 가이드라인을 함께 정리하면서, 그동안 쌓여 있던 부채를 한 번 정리해 보자는 생각으로 `FileTransporter`를 대대적으로 손봤습니다.

## 원래부터 밀려 있던 일들

이번에 정리한 내용을 보면 외부 YAML 설정 관리, 파일 브라우저와 업로드 흐름 재정리, 디렉터리 ZIP 다운로드 지원, 경로 검증 강화, Koin 기반 서비스 연결 정리, 프런트엔드와 백엔드 회귀 테스트 추가, Gradle 구성 정리 같은 것들이 포함됩니다.

겉으로 보면 여러 기능을 새로 만든 것처럼 보일 수도 있지만, 실제 감각은 조금 다릅니다. 대부분은 예전부터 필요하다고 느끼고 있었지만 손이 잘 가지 않았던 일들이었습니다. 다시 말해 "새로운 걸 만들었다"기보다, "밀려 있던 구조 정리와 기술 부채를 한꺼번에 처리했다"에 가깝습니다.

개인 프로젝트는 특히 이런 문제가 잘 생깁니다. 보이는 기능은 일단 돌아가니까 그대로 두고, 설정 구조나 책임 분리, 테스트 보강 같은 것은 언젠가 하겠다고 미뤄 두게 됩니다. 하지만 시간이 지나고 다시 코드를 보게 되면, 오히려 그런 부분이 발목을 잡습니다.

## 이번에는 왜 정리가 가능했나

이번에 가장 크게 달랐던 점은, 코딩 에이전트를 통해 "정리 작업에 들어가는 초기 비용"이 많이 낮아졌다는 것입니다.

설정을 YAML로 빼는 일, 화면 상태를 분리하는 일, 컴포넌트를 쪼개는 일, 테스트를 추가하는 일, 빌드 구성을 정리하는 일은 각각만 보면 아주 어려운 문제는 아닙니다. 다만 실제로는 기존 코드를 다시 읽고, 영향 범위를 보고, 바꾼 기준을 맞추고, 빠진 부분이 없는지 확인해야 해서 꽤 피곤한 작업이 됩니다. 이런 종류의 일은 그래서 늘 "나중에"로 밀리기 쉽습니다.

이번에는 그 부분에서 AI가 꽤 도움이 됐습니다. 기존 코드를 바탕으로 초안을 만들고, 정리 방향을 빠르게 비교해 보고, 빠진 테스트 포인트를 떠올리고, 리팩터링 단위를 먼저 잘게 나누는 데 유용했습니다.

물론 AI가 알아서 다 맞게 처리해 주는 것은 아닙니다. 무엇을 우선할지, 어디까지 바꿀지, 어떤 구조를 유지할지는 결국 제가 결정해야 했습니다. 그래도 적어도 "다시 들어가서 정리해야 하는데 너무 귀찮다"는 감각은 확실히 줄어들었습니다.

## 놓치기 쉬운 부분도 다시 보게 됐다

이번 작업에서 예상보다 좋았던 점은, 놓치기 쉬운 부분을 다시 보게 됐다는 것입니다. 예를 들어 경로 검증처럼 원래 중요하다고 알고 있던 부분도, AI와 함께 코드를 다시 읽고 수정 단위를 나누다 보면 "이 부분을 정말 충분히 봤나?"를 한 번 더 묻게 됩니다.

혼자 익숙한 코드만 보고 있으면 대충 넘어가기 쉬운 부분이 있습니다. 예전에 작성한 코드일수록 더 그렇습니다. 그런데 이번에는 그런 부분을 다시 점검하면서, 단순히 눈에 보이는 기능만이 아니라 안전성이나 예외 처리 같은 부분까지 한 번 더 확인할 수 있었습니다.

이건 생산성 이야기와는 조금 다른데, 개인적으로는 오히려 이런 쪽이 더 중요하게 느껴졌습니다. 빠르게 고치는 것보다, 전에 놓쳤던 걸 다시 보게 되는 것이 더 큰 가치일 수도 있으니까요.

## 구현 룰도 같이 다시 정리하게 됐다

또 하나 흥미로웠던 점은, AI를 리팩터링에 활용하다 보니 구현 가이드라인 자체를 다시 정리하게 됐다는 것입니다.

책임은 어디서 나눌지, 상태는 어느 레이어가 가져가야 할지, 테스트는 어느 단위까지 있어야 할지 같은 기준은 평소 머릿속에는 대충 있는데, 막상 오래된 프로젝트를 다시 손보려 하면 그 기준이 흐릿해져 있기도 합니다. 이번에는 코드 수정과 함께 `AGENTS.md` 같은 가이드라인도 함께 손보는 방향으로 생각하다 보니, 단순히 현재 코드를 고치는 데서 끝나지 않고 "앞으로는 이 프로젝트를 어떤 기준으로 유지할 것인가"를 다시 써 보게 됐습니다.

즉, AI는 여기서 단순한 자동화 도구라기보다, 구현 피드백을 되돌려 주는 상대 역할도 했습니다. 코드를 바꾸는 과정에서 왜 이렇게 나누는지, 왜 여기까지 테스트해야 하는지, 왜 이 룰이 필요한지를 다시 설명하게 만들었기 때문입니다.

## 그래서 실제로 바뀐 것

설정 쪽에서는 저장 루트 디렉터리와 업로드 최대 크기를 `config/application.yaml`로 뺄 수 있게 했고, 필요하면 시스템 프로퍼티로 덮어쓸 수 있게 했습니다. 이전보다 훨씬 운용하기 쉬운 형태가 됐습니다.

프런트엔드에서는 파일 브라우저를 나누어 정리하면서 업로드와 다운로드 흐름을 다시 손봤고, 밝은 테마와 다크 모드 전환도 추가했습니다. 헤더, 업로드 패널, 상세 모달, 전송 이력 같은 부분도 분리했습니다.

백엔드에서는 저장 루트 바깥으로 나가지 못하도록 경로 검증을 강화했고, 잘못된 대상에 대한 응답도 더 명확하게 정리했습니다. 동시에 Koin을 붙여 서비스 연결을 정리하고, 전송 상태와 서버 시작 책임도 나눴습니다.

그리고 무엇보다 프런트엔드와 백엔드 양쪽에 회귀 테스트를 추가할 수 있었던 것이 컸습니다. 이런 부분은 늘 필요하다고 생각하면서도, 따로 시간 내서 정리하기가 쉽지 않았기 때문입니다.

## 마지막으로

이번 `FileTransporter` 정리는 기능 몇 개를 더한 작업이라기보다, 오랫동안 방치해 두었던 앱을 다시 유지 가능한 형태로 되돌리는 작업에 더 가까웠습니다. 그리고 그 과정에서 AI는 단순히 코드를 대신 쓰는 도구가 아니라, 미뤄 두었던 정리에 착수하게 만들고, 놓치고 있던 부분을 다시 보게 하고, 구현 기준까지 재정비하게 만드는 도구로 작동했습니다.

아마 앞으로도 개인 프로젝트를 손볼 때는 이런 방식이 더 많아질 것 같습니다. 예전에는 시작하기도 전에 부담스러워서 미뤄 두던 일들이, 이제는 그래도 한 번 정리해 볼 만한 작업으로 느껴지기 때문입니다.
