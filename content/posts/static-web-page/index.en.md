It's a bit late, but as an engineer who has mainly worked on backends (especially from a position where Java is the main focus since Spring has taken over everything), when I look at the frontend world these days, I feel like I can instantly become an old-fashioned person even if I look away for a moment. With so many different libraries, frameworks, design philosophies, and platforms appearing and disappearing, it can be hard to know which ones to learn.

For example, when it comes to app design concepts, SPA[^1] is trending, but with the advent of [WASM](https://webassembly.org), PWA[^2] is also starting to appear. Among JavaScript frameworks, [Angular](https://angular.io), [React](https://reactjs.org), and [Vue.js](https://vuejs.org) are currently known as the "top three" because they are "fast because they use Virtual DOM," but there is also [Svelte](https://svelte.dev), which is "fast because they update the DOM directly." [Deno](https://deno.land) has appeared, saying that [Node.js](https://nodejs.org) failed, and [Dart](https://dart.dev), which tried to reach the current position of [TypeScript](https://www.typescriptlang.org) but seemed to have failed, seems to have made a comeback with [Flutter](https://flutter.dev) and [Fuchsia](https://fuchsia.dev).

In this way, the front-end industry is in a confusing situation with a flood of new concepts and technologies, but interestingly enough, it seems that there are also a fair number of concepts related to building web apps, or rather websites, that go against the trend. This is the topic of today's post, "Static Web Pages."

## What is a static web page?

Being "static" probably means that it has characteristics that can be classified as "dynamic." The most important criterion for whether it is dynamic or static here seems to be whether some processing is performed based on the input, or whether the data stored on the server is always returned as is. For example, Wikipedia defines it as follows:

> A static web page (sometimes called a flat page or a stationary page) is a web page that is delivered to the user's web browser exactly as stored, in contrast to dynamic web pages which are generated by a web application.

In other words, if a site is created as an "application," it can be said to be dynamic, or if it is just a "page" that displays fixed data, it can be said to be static.

## Why now?

The concept of returning data exactly as it was stored on a server existed before the advent of web applications. At the time, websites were hosted by uploading resources such as HTML, CSS, and images to a server using FTP, and this is the same method for static web pages today.

In other words, if you just look at the concept, static web pages are similar to things like "homepages" and "blogs" that were popular in the past. Why is it that static web pages, which seem to go against the times, are attracting more attention now?

Of course, unlike back then, with the development of JavaScript, it is now possible to perform simple processing in the browser without relying on a server, and if there is an appropriate API, there is no need to implement a backend. Services such as AWS Lambda and Azure Functions have created a serverless environment, and there are now more methods to choose from depending on the purpose of the frontend than the backend.

And then there is the most fundamental purpose of the site. For sites that don't require a lot of processing, such as blogs, portfolios, and event guides, static sites have advantages over dynamic web pages in terms of resource usage, cost, performance, and security. For example, like this blog right now.

Nowadays, there are many tools that can create static web pages based on documents written in Markdown, etc., and there are many ways to host a page by simply uploading the data generated with such tools. In a sense, it can be said that it is now the cheapest time to create static web pages because you can manage the source yourself and can easily host it.

## Tools for creating static web pages

In most cases, static website generation tools are not difficult to use. You need to build an execution environment for the base language and then download and apply a template, or create one, but to build an environment to run the tool, you can just follow the steps on the tool's homepage, and you can just bring in a template that someone else has made and modify it a little. If it's a simple fix, it's not difficult because you just need to modify the HTML, CSS, JSON, and YAML files (in some cases, you can select various options in the property file). All that's left to do is write the article using Markdown.However, from my experience, if you are going to make major changes to a ready-made template or start from scratch, it is better to base it on a language that you can use. For example, this blog was created using [Jekyll](https://jekyllrb.com), and I have modified some of the templates, but since I am not familiar with Ruby, I have not modified the fundamental parts.

In addition to Jekyll, there seem to be many tools based on Nodejs. For example, there are things like [Metalsmith](https://metalsmith.io) and [Hexo](https://hexo.io), as well as React-based [Gatsby](https://www.gatsbyjs.com) and Vue.js-based [VuePress](https://vuepress.vuejs.org). If you are comfortable using JavaScript and its frameworks, this may be for you. There is also [Hugo](https://gohugo.io), which is based on the Go language, so if you are familiar with Go, you may choose this.

## Platforms that can host static web pages

Once you have created a static site using a tool, you will need hosting. For hosting static web pages, there are services such as [GitHub Pages](https://pages.github.com) and [netlify](https://www.netlify.com), which are also used in this blog, and they have the convenience of automatically completing build and deployment when you upload data using Git etc.

Another method is to use the cloud. Cloud services such as AWS, Azure, and Google Cloud Platform always have storage services that can be accessed from the web, such as [S3](https://aws.amazon.com/s3) and [Blob](https://azure.microsoft.com/en-us/products/storage/blobs), but there is also a method of hosting a static site created with a tool by uploading it to this storage. Especially in the case of Google services, there is also a platform specialized for application development called [Google Firebase](https://firebase.google.com), so I think it might be a candidate to consider if you need to collaborate with an API.

## Finally

While I've just introduced (or rather mentioned) tools and platforms, there's a lot more you can do and use with static web pages. Even if it is the same static web page, the combination of technologies used to create something will vary considerably, but you can find out what can actually be created from the Examples tab of [JAMstack](https://jamstack.org), so please refer to it.

Personally, I'm interested in Gatsby, and while studying React, I'm thinking of using it to build a portfolio site someday. Oracle Cloud also provides free VM and DB, so it may be possible to create a simple API. No, it's a good time. Everyone is welcome to try building their own site using static web pages.

[^1]: Abbreviation for Single Page Application, which refers to a web application that uses a single screen to improve development efficiency and performance, whereas conventional web applications consist of multiple screens. Because the screen is separated from the server side, there is a wide range of backend choices.
[^2]: An abbreviation for Progressive Web Application, which refers to an application that is a web application but has the advantages of a native application (high performance, can be used offline by installing).
