주로 백엔드 일을 해 온, 그것도 Java와 Spring 중심으로 일해 온 입장에서 보면 요즘 프런트엔드 세계는 잠깐만 한눈을 팔아도 금세 뒤처질 것처럼 느껴집니다. 라이브러리와 프레임워크, 설계 방식, 플랫폼이 워낙 빠르게 바뀌다 보니, 오히려 무엇부터 배워야 할지 더 헷갈릴 때도 있습니다.

예를 들어 앱 설계 관점에서는 SPA[^1]가 트렌드이긴 하지만, [WASM](https://webassembly.org) 등장과 함께 PWA[^2]도 함께 주목받고 있습니다. JavaScript 프레임워크는 "Virtual DOM을 사용하니 빠르다"는 이유로 [Angular](https://angular.io), [React](https://reactjs.org), [Vue.js](https://vuejs.org)가 지금의 이른바 3강이 되었지만, "DOM을 직접 갱신하니 빠르다"는 [Svelte](https://svelte.dev)도 있습니다. [Node.js](https://nodejs.org)가 실패했다고 보는 사람들도 [Deno](https://deno.land)를 이야기하고, [TypeScript](https://www.typescriptlang.org)가 차지한 현재의 위치를 노렸지만 실패한 것으로 보였던 [Dart](https://dart.dev)도 [Flutter](https://flutter.dev)와 [Fuchsia](https://fuchsia.dev) 덕분에 다시 기회를 얻는 것처럼 보입니다.

이처럼 프런트엔드 쪽은 새로운 개념과 기술이 끊임없이 쏟아져 나옵니다. 그런데 흥미롭게도, 이런 흐름과는 반대로 보이는 방식 역시 꾸준히 살아남아 있습니다. 이번 글에서 이야기하려는 정적 웹 페이지가 바로 그런 예입니다.

## 정적 웹 페이지란

여기서 말하는 "정적"은 "동적"과 대비되는 개념입니다. 가장 단순하게 말하면, 요청마다 무언가를 계산해서 만들어 내느냐, 아니면 서버에 저장된 결과를 그대로 돌려주느냐의 차이라고 볼 수 있습니다. Wikipedia는 이를 다음처럼 설명합니다.

> A static web page (sometimes called a flat page or a stationary page) is a web page that is delivered to the user's web browser exactly as stored, in contrast to dynamic web pages which are generated by a web application.

즉 사이트가 애플리케이션처럼 동작하면 동적이고, 미리 만들어 둔 내용을 그대로 보여 주는 쪽에 가깝다면 정적이라고 할 수 있습니다.

## 왜 지금 다시 만드는가

서버에 저장된 데이터를 그대로 돌려주는 개념은 웹 애플리케이션이 등장하기 전부터 있었습니다. 당시 웹사이트는 HTML, CSS, 이미지 같은 리소스를 FTP 등으로 서버에 올려서 호스팅했는데, 이것이 지금의 정적 웹 페이지 방식과 비슷합니다.

개념만 보면 정적 웹 페이지는 예전의 "홈페이지"나 단순한 "블로그"에 가깝습니다. 그런데 왜 이런 방식이 지금 다시 주목받을까요?

이제는 JavaScript의 발전 덕분에 간단한 처리는 서버가 아니라 브라우저에서 처리할 수 있고, 적절한 API만 있으면 백엔드를 꼭 직접 구현하지 않아도 됩니다. AWS Lambda나 Azure Functions 같은 서비스 덕분에 서버리스 환경도 쉽게 쓸 수 있게 되었습니다. 그래서 백엔드보다 프런트엔드 쪽 목적에 따라 선택할 수 있는 방법이 더 많아졌습니다.

그리고 더 근본적인 이유는 사이트의 목적 자체에 있습니다. 블로그, 포트폴리오, 이벤트 안내처럼 애초에 복잡한 처리가 필요 없는 사이트라면, 정적으로 만드는 편이 리소스 사용량, 비용, 성능, 보안 면에서 동적 웹 페이지보다 유리합니다. 예를 들면 지금 이 블로그처럼요.

최근에는 Markdown 같은 문서를 기반으로 정적 웹 페이지를 만들어 주는 도구도 많습니다. 직접 소스를 관리하면서 손쉽게 배포할 수 있으니, 어쩌면 지금이야말로 정적 웹 페이지를 만들기 가장 쉬운 시기인지도 모르겠습니다.

## 정적 웹 페이지를 만드는 도구

대부분의 경우 정적 사이트 생성 도구를 쓰는 일은 어렵지 않습니다. 기반 언어의 실행 환경을 준비하고, 템플릿을 내려받아 적용하거나, 직접 만들어 가면 됩니다. 실행 환경은 도구 문서대로 따라 하면 되고, 템플릿도 이미 만들어진 것을 가져와 조금 손보면 충분합니다. 간단한 수정이라면 HTML, CSS, JSON, YAML 정도만 다뤄도 됩니다. 그 뒤에는 Markdown으로 글만 써 나가면 됩니다.

다만 제 경험상, 기존 템플릿을 크게 바꾸거나 아예 새로 만들려면 내가 잘 다룰 수 있는 언어를 기반으로 한 도구를 선택하는 편이 좋습니다. 예를 들어 이 블로그는 [Jekyll](https://jekyllrb.com)로 만들었고 템플릿도 일부 수정했지만, Ruby에 익숙하지 않아서 근본적인 부분은 많이 건드리지 않았습니다.

Jekyll 말고는 Node.js 기반 도구가 많은 편입니다. 예를 들어 [Metalsmith](https://metalsmith.io), [Hexo](https://hexo.io) 같은 도구가 있고, React 기반의 [Gatsby](https://www.gatsbyjs.com), Vue.js 기반의 [VuePress](https://vuepress.vuejs.org)도 있습니다. JavaScript와 그 프레임워크에 익숙하다면 이쪽이 잘 맞을 수 있습니다. Go 기반의 [Hugo](https://gohugo.io)도 있으니, Go에 익숙한 사람이라면 이쪽도 좋은 선택입니다.

## 정적 웹 페이지를 올릴 플랫폼

도구로 정적 사이트를 만들었다면 이제 호스팅할 차례입니다. 정적 웹 페이지를 호스팅할 수 있는 서비스로는 이 블로그에서도 쓰는 [GitHub Pages](https://pages.github.com)나 [Netlify](https://www.netlify.com) 같은 서비스가 있습니다. Git 저장소에 올리기만 하면 자동으로 빌드와 배포를 해 주는 점이 편리합니다.

클라우드를 활용하는 방법도 있습니다. AWS, Azure, Google Cloud Platform 같은 서비스에는 [S3](https://aws.amazon.com/s3)나 [Blob](https://azure.microsoft.com/ko-kr/services/storage/blobs/)처럼 웹에서 접근할 수 있는 스토리지 서비스가 있습니다. 여기에 정적 사이트를 올려서 호스팅할 수도 있습니다. 특히 Google 쪽은 [Google Firebase](https://firebase.google.com) 같은 앱 개발용 플랫폼도 있어서, API 연계가 필요한 경우 고려해 볼 만합니다.

## 마지막으로

여기서는 도구와 플랫폼을 간단히 소개하는 정도로 끝냈지만, 정적 웹 페이지로 할 수 있는 일은 생각보다 많습니다. 같은 정적 웹 페이지라도 어떤 기술을 조합하느냐에 따라 결과물이 꽤 달라지는데, 실제 예시는 [JAMstack](https://jamstack.org)의 Examples 탭에서 많이 찾아볼 수 있습니다.

개인적으로는 Gatsby에 관심이 있어서, React 공부를 겸해 언젠가 포트폴리오 사이트를 하나 만들어 보고 싶습니다. Oracle Cloud처럼 무료 VM과 DB를 제공하는 서비스도 있으니, 필요하다면 간단한 API를 함께 붙이는 것도 가능하겠지요. 정적 웹 페이지를 시작하기에는 꽤 좋은 시대라고 생각합니다.

[^1]: Single Page Application의 약자로, 기존 웹 애플리케이션이 여러 화면으로 구성되던 방식과 달리 화면을 하나로 두고 개발 효율과 성능을 높이려는 웹 애플리케이션을 말합니다. 화면이 서버 사이드와 분리되어 있어 백엔드 선택지가 넓은 편이라는 특징도 있습니다.
[^2]: Progressive Web Application의 약자로, 웹 앱이면서도 네이티브 앱의 장점(높은 성능, 설치 후 오프라인 사용 가능 등)을 일부 가져온 애플리케이션을 말합니다.
