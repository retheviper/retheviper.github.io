This time, I would like to explain how to execute a Job generated by Jenkins. Although it is basically possible to execute jobs using the Jenkins management console, it is not convenient to log in and start jobs each time. So you start thinking, ``I want to automatically execute a Job under certain conditions.'' For example, one of the easiest things to think of is "periodic execution." This kind of setting is necessary for batch-like processing that is automatically executed by specifying a specific time or frequency. You may also want to execute something only when certain conditions are met. For example, I would like to build only when a commit occurs in a Git repository.

Jenkins Job allows you to set automatic job execution based on conditions like this. This is called `Build Trigger`. The basic settings have the minimum required patterns, and depending on the combination of plug-ins, you can configure Jobs to run with more diverse settings. In this post, I would like to explain what build triggers are and how they can be used.

## Type of Build Trigger

When you generate a Job in Jenkins, you can see that after the General and Source Code Management tabs there is a tab called Build Trigger. The number may increase or decrease depending on the plug-in configuration, so please check it once. I chose the recommended plugin configuration when installing Jenkins, so I have the following configuration.

![Jenkins Build Triggers](jenkins_build_triggers.webp)

- Trigger builds remotely
- Build after other projects are built
- Build periodically
- GitHub hook trigger for GITScm polling
- Poll SCM

Now, let's take a look at what each of these functions means and how they can be used.

## Trigger builds remotely

![Jenkins Build Triggers Remote](jenkins_trigger_remote.webp)

As the expression "remote" suggests, this means that the Job can be executed externally without connecting to the Jenkins management console. If you select this menu, you will be able to call a Job using a URL. If you enter the token name used for authentication in `Authentication Token`, you can call the Job with the URL below.

(Assuming Jenkins URL is `192.168.0.2:8080`, Job name is `TriggerTest`, and token name is `buildToken`)

1. `http://192.168.0.2:8080/job/TriggerTest/build?token=buildToken`
2. `http://192.168.0.2:8080/job/TriggerTest/buildWithParameters?token=buildToken`

The first setting is used when simply executing the Job without any parameters. The second case can be used when you want to pass parameters. The parameters received here can be set by checking `This project is parameterized` on the General tab and adding parameters. The values received as parameters can be used as external variables in shell scripts within the Job.

![Jenkins Build Parameter](jenkins_build_parameter.webp)

This is a useful feature when you need to make calls from outside, such as in Linux's Curl.

## Build after other projects are built

![Jenkins Build After](jenkins_build_after.webp)

Jenkins allows you to create multiple jobs. And generating multiple Jobs means dividing the Jobs, each with a different function, according to their purpose. However, depending on the case, it may be necessary to link these jobs. Even if methods and classes are separated within the same program, they will be used in conjunction.

If you check this option, you can enter what Job you want to build after. You can also register multiple jobs. Enter the Job name you want to build first in `Projects to watch` and select the three options below. There are three levels of Jenkins Job execution results: `stable` (success), `unstable` (partial success), and `fail` (failure). What you choose here is the condition for determining which of the three results the previous job achieved.

- Trigger only if build is stable
- Trigger even if the build is unstable
- Trigger even if the build fails

The first case is when the previous job was successfully built. If you need a deliverable, this is an option. The second option is to run even if the build is unstable. The third option is executed even if it fails. Each one has a different purpose and use, so choose the one that suits your needs.

## Build periodically![Jnekins Build Period](jenkins_build_period.webp)

Use this option if you want periodic execution based on time. If you write a cycle in the `Schedule` column according to Jenkins-specific rules, the Job will be automatically built repeatedly according to the conditions. It's quite complicated, but I'll explain it to you.

`Basic format`

1. Divide time units based on blank spaces.
2. The entry order is minute, hour, day of month, month, week.
    1. Minute: number from 0 to 59
    2. Hour: number from 0 to 23
    3. Day of month: number from 1 to 31
    4. Month: number from 1 to 12
    5. Week: Number from 0 to 7 (0 and 7 are days, 1 is Monday, 2 is Tuesday, 3 is Wednesday, 4 is Thursday, 5 is Friday, 6 is Saturday)
3. `*` means that it accepts the entire range of values that can be entered (all numbers from 0 to 59 when entered in the minute position)
4. You can specify a range by including `-` like `M-N` (`2-4` is a value from 2 to 4)
5. `M-N/X` or `*/X` means interval (entering `*/2` in the time position means every two hours)
6. Multiple values can be enumerated like `A,B,C`

In addition, if there are multiple Jobs that are executed periodically, we recommend using `H`. For example, if there are 10 types of Jobs and each is set to `0 0 * * *`, all Jobs will be executed every day at 0:00. If the server specs are not very good or the number of jobs is larger than that, the load will be at a level that cannot be ignored.

Therefore, by specifying `H H * * *`, Jenkins will automatically make a decision and execute the Job during times when there is less load on the server. Of course, you can also specify a range with `H`, so if you specify something like `H H(0-1) * * *`, the build will be performed every day from 0:00 to 1:00, during times when the load is low.

Let's look at some examples written following this format.

- Run build every 15 minutes: `H/15 * * * *`
- Run a build every 10 minutes by the 30th minute of every hour (1:00 to 1:30, 2:00 to 2:30, etc.): `H(0-29)/10 * * * *`
- Build runs every 2 hours between 9:00 and 16:00 on weekdays: `H 9-16/2 * * 1-5`
- Build execution only on the 1st and 15th from January to November: `H H 1,15 1-11 *`

It seems like it will take some time to get used to the format, but it is an attractive option because it can handle even quite complex rules.

## GitHub hook trigger for GITScm polling

![Jenkins GitHub Hook](jenkins_github_hook.webp)

This option works in conjunction with [GitHub](https://github.com) and means that a build will occur when a push occurs to the repository. There is no complicated configuration in Jenkins, just select this option, but you will need to configure it to notify Jenkins that a push has occurred in the GitHub repository.

When you enter the GitHub repository, there is a tab at the top called `Settings`. Click on it and further enter the menu called `Webhooks`. Then a screen like the one below will appear.

![GitHub Webhook](github_webhook1.webp)

If you press the `Add Webhooks` button here, the following screen will appear again.

![GitHub Webhook 2](github_webhook2.webp)

There is no need for complicated settings here either, just write Jenkins URL/github-webhook in `Payload URL`. For example, if the Jenkins URL is `192.168.0.2:8080`, it will be written as `http://192.168.0.2:8080/github-webhook/`. Now, when a push occurs to the repository, the Job will be automatically built. This also makes it easy to manage Git on your web server.

## Poll SCM

![Jenkins Poll SCM](jenkins_poll_scm.webp)'

SCM stands for `Source Control Management`. [^1] It refers to tools such as Git and SVN. When you select Poll SCM, an input field for `Schedule` appears, similar to `Build periodically`. Enter the polling interval there as well. The difference is that the build does not necessarily start at the set time. At the set time, the source is checked and the build runs only if there are changes. It seems to be a setting that works well together with Git and SVN.
