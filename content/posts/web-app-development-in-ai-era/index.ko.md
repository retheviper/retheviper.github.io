요즘 웹 애플리케이션 개발의 전제가 조금씩 바뀌고 있다고 느낍니다. 특히 큰 변화는 AI가 단순히 코드를 작성해 주는 도구가 아니라, 실제 조작 자체를 맡을 수 있는 존재가 되어 가고 있다는 점입니다. MCP 나 Agent 같은 흐름을 보고 있으면, 앞으로의 웹 앱은 "사람이 브라우저에서 모든 것을 조작하는 것"이라기보다 "서버 위의 상태를 UI 와 AI 가 함께 다루는 것"으로 보는 편이 더 자연스러워질 것 같습니다.

결론부터 말하면, 앞으로의 웹 앱 개발은 **"프론트엔드 중심"에서 "서버 중심 + AI 조작"으로 이동한다** 고 생각합니다.

## UI 의 역할은 달라진다

그동안 UI 는 조작과 표시를 모두 담당하는 것이 일반적이었습니다. 버튼을 누르고, 폼을 입력하고, 화면을 이동하고, 그 결과를 다시 보여 주는 장소였죠. 말하자면 UI 는 사람이 앱을 실제로 만지는 공간 그 자체였습니다.

하지만 앞으로는 조작의 일부, 혹은 꽤 큰 부분을 AI 가 맡게 될 것입니다. 사람이 여러 화면을 오가며 직접 수행하던 작업을 Agent 가 한 번에 처리하는 장면은 점점 많아질 것입니다. 그렇게 되면 UI 에 남는 핵심 역할은 조작 그 자체보다 "지금 무엇이 일어나고 있는가를 보여 주는 것"으로 옮겨갑니다.

즉, UI 는 "조작의 장소"에서 "가시화의 장소"로 바뀌게 됩니다.

## 프론트와 백의 분리 방식도 달라진다

기존의 대표적인 구조는 SPA 프론트엔드와 그 뒤의 REST API 였습니다. 물론 이 구성이 당장 사라지지는 않겠지만, 최근에는 풀스택 프레임워크와 Server Components, Server Actions 같은 방식이 훨씬 자연스러운 선택지가 되고 있습니다.

이 흐름에서는 프론트엔드와 백엔드의 경계가 "HTTP API 를 얼마나 명시적으로 자르느냐"가 아니라 "상태와 책임을 어디에 두느냐"로 결정됩니다. 그 결과 예전처럼 REST API 를 촘촘하게 잘게 나누기보다, 서버 쪽 로직을 그대로 애플리케이션의 중심에 두는 경우가 늘어날 것입니다.

## 렌더링 전략은 혼합이 기본이 된다

렌더링도 이제는 SSG, SSR, CSR 중 하나를 고르는 문제가 아니라고 생각합니다.

- SSG 는 정적 콘텐츠에 적합하고
- SSR 은 사용자 의존 데이터나 초기 표시가 중요한 부분에 적합하며
- CSR 은 정말 복잡한 UI 에만 집중해서 쓰는 방식이 될 것입니다

중요한 것은 무엇이 정답인지 하나를 고르는 일이 아니라, 이것들을 자연스럽게 섞는 것입니다. 페이지 전체를 처음부터 끝까지 전부 CSR 로 가져갈 필요는 줄어들고, 반대로 전부 정적으로 만들면 되는 것도 아닙니다. 앞으로는 "선택"보다 "혼합"이 기본이 될 것입니다.

## JavaScript 는 사라지지 않지만 역할은 줄어든다

JavaScript 자체가 필요 없어지지는 않을 것입니다. 웹 UI 가 브라우저에서 동작하는 한 런타임은 계속 필요합니다.

다만 그 역할은 지금보다 얇아질 가능성이 큽니다. 지금까지처럼 화면 쪽에 많은 상태와 비즈니스 로직을 끌어안고, 앱의 본체가 사실상 프론트엔드에 있는 구조는 점점 줄어들 것입니다. 앞으로의 JavaScript 는 UI 제어를 위한 얇은 층으로 남고, 애플리케이션의 본체는 더 서버 쪽으로 이동할 것입니다.

즉, "JavaScript 가 사라진다"가 아니라, "많은 JavaScript 를 직접 써야 할 필요가 줄어든다"는 변화에 가깝습니다.

## AI 가 바꾸는 것은 구현만이 아니다

AI 가 바꾸는 것은 구현 속도만이 아닙니다. 구현, 수정, 테스트 보조뿐 아니라, 애플리케이션의 조작 자체도 대신할 수 있게 됩니다.

그 대신 사람의 가치가 더 커지는 부분은 따로 있습니다. 설계, 구조, 책임 분리, 제약 정의 같은 것들입니다. AI 가 코드를 많이 써 줄 수 있기 때문에 오히려 "무엇을 어떤 구조로 둘 것인가"를 먼저 정하는 일이 더 중요해집니다.

앞으로는 코드를 직접 쓰는 행위보다, 코드가 올라갈 구조를 정하는 일이 더 중요해질 것이라고 생각합니다.

## React 는 남겠지만 더 이상 주인공은 아닐 수 있다

React 가 사라질 것이라고 생각하지는 않습니다. 앞으로도 UI 를 구성하는 데 중요한 런타임으로 남을 가능성이 큽니다.

다만 예전처럼 "애플리케이션의 중심 그 자체"로 취급되는 경우는 줄어들 수 있습니다. React 는 UI 를 표현하는 레이어 중 하나로 수렴하고, 중심은 다시 서버 위의 상태와 로직으로 이동할 것입니다. 저는 이 흐름이 꽤 현실적이라고 봅니다.

## 새로운 아키텍처는 API 의 의미도 바꾼다

기존 구조는 대체로 `UI -> API -> Server -> DB` 였습니다.

앞으로는 `AI / UI -> Server Logic -> DB` 라는 그림이 더 실제에 가까워질 것입니다. 물론 외부 공개 API 나 시스템 간 연동을 위한 API 는 앞으로도 필요합니다. 하지만 애플리케이션 내부의 중심 구조로 보면, API 를 잘게 자르는 것보다 서버 로직을 얼마나 직접적이고 자연스럽게 둘 수 있는지가 더 중요해집니다.

그 의미에서 API 는 "사람이 프론트엔드에서 호출하는 인터페이스"에서 "AI 나 다른 시스템이 사용하는 도구"로 성격이 바뀌어 갈 것입니다.

## 앞으로 중요해질 설계

이 흐름에서 특히 중요해질 설계는 다음과 같다고 생각합니다.

### 도구 설계

AI 가 사용할 것을 전제로 한다면 도구의 단위는 너무 잘지 않은 편이 좋습니다. 무엇을 하기 위한 동작인지 의도가 명확해야 합니다.

### 상태 관리

상태는 가능한 한 서버를 단일 진실로 두는 편이 좋습니다. 프론트엔드에 복잡한 진실을 들고 있을수록 UI 와 AI 양쪽에서 다룰 때 정합성이 깨지기 쉽습니다.

### 타입과 도메인

애매함이 적은 도메인 모델이 더 중요해집니다. enum 이나 value object 같은 방식으로 의미를 닫아 두고, 잘못된 상태를 만들기 어렵게 하는 설계는 AI 시대에도 매우 유효합니다.

### 관측 가능성

AI 가 조작을 담당하게 되면, 나중에 무슨 일이 일어났는지 추적할 수 있어야 합니다. 로그, 이벤트, 감사 기록 같은 관측 가능성은 단순한 운영 보조가 아니라 신뢰성의 핵심 요소가 됩니다.

## 약해질 것들

반대로 점점 설득력이 약해질 것들도 있습니다.

- 무거운 SPA 전제 설계
- 과도한 프론트엔드 상태 관리
- 지나치게 잘게 나뉜 REST API

이것들이 완전히 사라지지는 않겠지만, "일단 이렇게 하는 게 당연하다"는 식의 기본값으로 두기 어려워질 것이라고 봅니다.

## 마무리

앞으로의 웹은 단순한 브라우저 앱이라기보다, **"서버 위의 상태를 AI 와 UI 로 조작하고 가시화하는 시스템"** 에 더 가까워질 것입니다.

그 안에서는 프론트엔드를 두껍게 만드는 것보다, 서버 쪽의 상태, 책임, 도구, 관측 가능성을 어떻게 설계하느냐가 훨씬 더 중요해집니다. AI 가 코드를 쓰는 시대라기보다, AI 가 시스템의 사용자이기도 한 시대라는 관점에서 웹 애플리케이션을 다시 봐야 하지 않을까 생각합니다.
