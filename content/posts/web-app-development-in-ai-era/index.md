最近、Webアプリ開発の前提が少しずつ変わってきていると感じています。特に大きいのは、AIが単にコードを書く補助ではなく、操作そのものを引き受ける存在になりつつあることです。MCP や Agent のような仕組みを見ていると、これからのWebアプリは「人がブラウザ上で全部操作するもの」として作るより、「サーバ上の状態を、UI と AI の両方から扱うもの」として考えた方が自然になっていく気がします。

結論から言うと、これからの Web アプリ開発は **「フロントエンド中心」から「サーバ中心 + AI操作」へシフトする** と思っています。

## UI の役割は変わる

これまでのUIは、操作と表示の両方を担うのが普通でした。ボタンを押し、フォームを入力し、画面遷移を作り、その操作結果を表示する。つまり UI は「人がアプリを触るための場所」そのものでした。

ただ、これからは操作の一部、あるいはかなり大きな部分を AI が担うようになるはずです。人が細かく画面をたどって実行していた作業を、Agent がまとめて処理する場面は増えていくでしょう。そうなると UI に残る中心的な役割は、操作そのものよりも「今何が起きているかを見せること」に寄っていきます。

つまり UI は、「操作の場」から「可視化の場」へ変わっていくと思います。

## フロントとバックの分け方も変わる

これまでの典型は、SPA をフロントに置き、その背後に REST API を用意する構成でした。もちろんこの形がすぐ消えるわけではありませんが、最近はフルスタックフレームワークや Server Components、Server Actions のような仕組みがかなり自然な選択肢になってきました。

この流れでは、フロントエンドとバックエンドの境界は「HTTP API を明示的に切ること」ではなく、「どこに状態と責務を置くか」で決まるようになります。結果として、以前ほど REST API を細かく分けず、サーバ側のロジックをそのままアプリの中心に置くケースは増えていくはずです。

## レンダリング戦略は混在が前提になる

レンダリングも、SSG か SSR か CSR かを一つ選ぶ時代ではなくなっていくと思います。

- SSG は静的コンテンツ向け
- SSR はユーザー依存や初期表示重視の部分向け
- CSR は本当に複雑な UI の部分向け

大事なのは、どれが正しいかを一つに決めることではなく、それぞれを素直に混在させることです。ページ全体を最初から最後まで全部 CSR で持つ必要は減り、逆に全部を静的に寄せればよいわけでもありません。今後は「選択」より「混在」が前提になるはずです。

## JavaScript は消えないが、役割は縮小する

JavaScript 自体が不要になるとは思っていません。Web の UI を動かす以上、ブラウザ上のランタイムはこれからも必要です。

ただ、その役割は今より薄くなると思います。これまでのように、画面側に大量の状態やビジネスロジックを抱え込み、アプリの本体がほぼフロントエンドにあるような構成は減っていくでしょう。これからの JavaScript は、UI 制御のための薄い層として残り、アプリケーションの本体はよりサーバ側に寄っていくはずです。

つまり、「JavaScript が消える」のではなく、「JavaScript を大量に書く必要が減る」という変化です。

## AI が変えるのは実装だけではない

AI が変えるのは実装速度だけではありません。実装、修正、テスト補助だけでなく、アプリの操作自体も肩代わりできるようになります。

一方で、人間側の価値は別のところに寄ります。重要になるのは、設計、構造、責務分離、制約定義です。AI がコードを大量に書けるからこそ、「何をどの構造で持つべきか」を先に決めることの重みはむしろ増えます。

これからは、コードを書くことそれ自体よりも、コードが乗る構造の方が重要になっていくと思います。

## React は残るが、主役ではなくなる

React が消えるとは思いません。今後も UI を組み立てるための有力なランタイムであり続けるはずです。

ただ、以前のように「アプリケーションの中心そのもの」として扱う場面は減るかもしれません。React はあくまで UI を表現するためのレイヤーの一つに寄っていき、主役はサーバ上の状態とロジックに戻っていく。そういう流れになる可能性はかなり高いと見ています。

## 新しいアーキテクチャは API の意味も変える

従来の構成は、だいたい `UI -> API -> Server -> DB` でした。

これからは、`AI / UI -> Server Logic -> DB` という見方の方が実態に近くなるはずです。もちろん外部公開APIやシステム間連携のための API は今後も必要です。ただ、アプリ内部の中心線としては、API を細かく切ることより、サーバロジックをどれだけ素直に持てるかの方が重要になります。

その意味で API は、「人間がフロントエンドから叩くためのもの」から、「AI や他のシステムが使うツール」へと意味が変わっていくでしょう。

## これから重要になる設計

この流れの中で、特に重要になるのは次のような設計だと思います。

### ツール設計

AI に触らせる前提なら、ツールの粒度は細かすぎない方がよいです。何をするための操作なのかが明確で、意図が読み取りやすいことが重要になります。

### 状態管理

状態はできるだけサーバを唯一の真実に寄せた方がよいと思います。フロント側に複雑な真実を持つほど、UI と AI の両方から扱う時に整合性が崩れやすくなります。

### 型とドメイン

曖昧さの少ないドメインモデルがより重要になります。enum や value object のような形で意味を閉じ込め、無効な状態を作りにくくする設計は、AI が触る前提でもかなり効いてきます。

### 観測性

AI が操作するようになると、何が起きたのかを後から追えることが今まで以上に重要になります。ログ、イベント、監査のような観測性は、単なる運用補助ではなく、アプリの信頼性そのものになります。

## 消えていくもの

逆に、弱くなっていくものもあります。

- 重い SPA 前提の設計
- 過剰なフロントエンド状態管理
- 細かく分割されすぎた REST API

これらが完全になくなるとは言いませんが、「とりあえずそうしておく」が通りにくくなるのは確かだと思います。

## まとめ

これからの Web は、単なるブラウザアプリというより、**「サーバ上の状態を AI と UI で操作し、可視化するシステム」** に近づいていくはずです。

その中では、フロントエンドを厚く作ることより、サーバ側の状態、責務、ツール、観測性をどう設計するかの方がずっと重要になります。AI がコードを書く時代というより、AI がシステムの利用者にもなる時代として Web アプリを見直す必要があるのではないかと思っています。
